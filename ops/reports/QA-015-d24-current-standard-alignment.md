# QA-015 D-24 현행 안전조치 기준 정합성 보완

- 작업 ID: `QA-015`
- 기준일: 2026-07-22 (Asia/Seoul)
- 결과: **PASS 후보 — D-24 보완 결정안 작성 완료 / 실제 개인정보 처리와 제품 적용은 별도 승인 필요**
- 변경 범위: 이 보고서 1개 신규 작성
- 실행하지 않은 것: 실제 사업자등록번호·관계 서류·사진·후기 증빙 수집·조회·저장, 제품 코드·DB·API·RLS·Storage·환경변수·패키지·CHG-A~C 변경, 외부 연락·게시·배포

## 1. 총괄 결론

QA-013의 최소수집 방향은 유지할 수 있다. 다만 D-24 승인안은 아래 네 가지를 바로잡은 이 보고서를 기준으로 해야 한다.

1. 안전조치 기준은 **개인정보보호위원회고시 제2026-9호**를 현행 기준으로 사용한다.
2. 개인정보처리시스템의 **접근권한 부여·변경·말소 이력은 최소 3년** 보관한다. QA-013의 `access grant·revision 감사 1년` 제안은 이 법정 이력에 적용하지 않는다.
3. 개인정보처리시스템 **접속기록은 최소 1년**, 법정 조건에 해당하면 **최소 2년** 보관한다. 관계·사진·후기 증빙 원본의 `결정 후 30일 / 수집 후 최대 90일`과 섞지 않는다.
4. OPS-008의 사업자번호 암호화 원문·검색용 keyed hash 구안은 폐기한다. 사업자번호는 상태조회 요청 중 서버 메모리에서만 사용하고 응답 또는 오류 처리 직후 즉시 폐기한다. 원문·암호문·hash·token·retry payload를 영구 저장하지 않는다.

따라서 최신 추천 묶음은 다음과 같다.

`D24-1 추천 + D24-2 추천 + D24-3 추천 + D24-4 추천 + D24-5 추천 + D24-6 R-A + D24-7 추천 + D24-8 추천`

이 묶음의 승인은 **정책 방향 승인**이다. 실제 수집, API 호출, DB/RLS/Storage 구현, 개인정보처리방침·동의 문구 확정, 운영 배포를 승인하지 않는다.

### 2.2 제5조와 제8조의 적용 경계

- **접근권한 이력 3년**: 누가 개인정보처리시스템 권한을 부여받았고, 언제 변경·말소됐는지를 기록한 자료다. 업체의 페이지 소유권 판정 원본이나 관계 증빙 원본이 아니다.
- **접속기록 1년 또는 2년**: 개인정보처리시스템에 접속해 수행한 업무를 식별자·접속일시·접속지·처리 대상·수행업무 등으로 기록한 자료다. 원본 사업자번호나 증빙 사본을 로그에 복제하지 않는다.
- **2년 조건**: ① 5만명 이상 정보주체의 개인정보를 처리하는 시스템, ② 고유식별정보 또는 민감정보를 처리하는 시스템, ③ 관련 법에 따라 등록·신고한 기간통신사업자 중 하나에 해당하는 경우다.
- 손품해방이 어느 조건에 해당하는지는 실제 처리 항목·규모·사업자 지위를 확정한 뒤 판단한다. 판단 전에는 실제 민감 입력을 켜지 않고 `APPROVAL_REQUIRED`로 둔다.

### 2.3 2026-09-11 재확인 게이트

법률 제21445호는 2026-07-22 현재 아직 시행 전이다. 이 개정은 대표자·개인정보 보호책임자 책임, 침해 대응과 제재 등 운영 책임을 강화한다. 이 보고서에서 확인한 고시 제5조의 3년 및 제8조의 1년·조건부 2년 기준을 바꾸는 근거로 해석하지 않는다.

다음 두 시점에 국가법령정보센터와 개인정보보호위원회에서 다시 확인한다.

1. 실제 개인정보 수집·상태조회·증빙 업로드 기능을 구현하거나 배포하기 직전
2. 2026-09-11 이후 개인정보를 처리하기 직전

예정법, 시행령·고시, 적용 대상, 처리방침·책임자·사고 대응 계약이 정합하지 않으면 제품 적용을 중단한다.

## 3. QA-013·OPS-008 대체 관계

### 3.1 유지하는 QA-013 원칙

- 비민감 업체 등록을 먼저 받고 민감 입력은 필요한 단계로 늦춘다.
- 사업자 상태, 담당자 관계, 페이지 소유권, 편집 권한, 필드 검수, 공개, 문의 활성은 서로 독립 판정한다.
- 초기에는 식별 가능한 인물·미성년자 사진과 후기 증빙을 받지 않는다.
- 관계 확인은 공식 도메인 이메일·공개 대표번호 회신 같은 공식 업무 채널을 우선한다.
- 제한 원본은 case 단위 최소권한, 짧은 signed access, 마스킹 목록, 다운로드·export 기본 거부를 적용한다.
- 철회·파기·legal hold는 자료별로 분리하고 포괄 보존하지 않는다.

### 3.2 정정·폐기하는 부분

| 기존 문구·구안 | 최신 판정 | 후속 계약에서 사용할 문구 |
| --- | --- | --- |
| QA-013의 접근 grant·revision 감사 `종료 후 1년` | **정정** | 개인정보처리시스템 접근권한 부여·변경·말소 이력은 최소 3년. 별도 운영 결정 기록과 분리 |
| QA-013의 민감 증빙 접근·보안 로그 `1년 추천` | **조건 추가** | 접속기록은 최소 1년, 제8조 조건에 해당하면 최소 2년 |
| QA-013의 관계·사진·후기 증빙 원본 `결정 후 30일 / 수집 후 최대 90일` | **유지 가능한 운영 추천안** | 법정 고정기간이 아니며 실제 수집 전 처리 근거·예외·백업·수탁 계약을 법률/개인정보 책임자가 확인 |
| OPS-008의 사업자번호 `암호화 원문 참조 + 검색용 keyed hash` | **폐기** | 요청 중 메모리에서만 사용하고 응답·오류 직후 즉시 폐기. 원문·암호문·hash/token 저장 금지 |
| QA-013의 사업자번호 즉시 폐기 원칙 | **최종 우선안으로 유지** | DB, Storage, queue, retry payload, 로그, 분석, export, backup 어디에도 번호·파생 식별자를 남기지 않음 |

`암호화`와 `hash`는 사업자번호를 보존해도 된다는 허가가 아니다. 다른 제한 원본에 대한 암호화 설계는 별도 검토할 수 있지만 사업자번호 영구 저장 금지에는 예외를 만들지 않는다.

## 4. 기간별 데이터 구분 — 최신 R-A

아래에서 `운영 추천`은 법정 고정기간을 뜻하지 않는다. 실제 적용 전에 개인정보/법률 책임자가 처리 목적과 다른 법령의 예외를 확인해야 한다.

| 자료 | 보유 기준 | 성격 | 함께 저장하면 안 되는 것 |
| --- | --- | --- | --- |
| 사업자번호 일시 입력 | 상태조회 응답 또는 오류 처리 직후 즉시 폐기 | 최종 추천·영구 저장 금지 | 원문, 암호문, 일부 값, hash/token, retry payload, URL·로그·분석 값 |
| 관계 서류·사진 권리 서류·후기 증빙 원본 | 결정·철회 후 30일, 단 수집 후 최대 90일 | `R-A` 운영 추천 | 접근권한 3년 또는 접속기록 1/2년을 이유로 원본을 연장 보관하지 않음 |
| business·relationship·ownership 결정 기록 | 관계/claim 종료 후 1년 | 원문 없는 내부 운영 추천 | 증빙 사본, 사업자번호, 개인 연락처 복제 금지 |
| 접근권한 부여·변경·말소 이력 | 최소 3년 | 현행 고시 제5조 기준 | 증빙 원본·사업자번호를 이력 payload에 넣지 않음 |
| 개인정보처리시스템 접속기록 | 최소 1년, 제8조 조건 해당 시 최소 2년 | 현행 고시 제8조 기준 | 원문 사업자번호·증빙 본문·업로드 파일 복제 금지 |
| 삭제 증명 | 삭제 완료 후 1년 | 원문 없는 내부 운영 추천 | 삭제된 자료의 내용 재기록 금지 |
| backup | 30일 이내 순환 삭제 | 내부 운영 추천 | 사업자번호를 backup 대상에 포함하지 않음; 복원 시 deletion tombstone 재적용 |

### R-A와 대안 R-B

- **추천 R-A**: 위 표처럼 다른 증빙 원본을 결정·철회 후 30일, 수집 후 최대 90일로 제한한다.
- **대안 R-B**: 사업자번호를 제외한 다른 원본을 case 종료 후 90일, 수집 후 최대 180일까지 보유한다. 늦은 이의 대응은 쉬워지지만 침해 범위와 정당화 부담이 커 초기에는 추천하지 않는다.
- 두 안 모두 접근권한 이력 최소 3년과 접속기록 최소 1년·조건부 2년을 별도 적용한다.
- 보유기간을 확정하지 못하면 해당 원본 수집을 시작하지 않는다. 무기한 보유는 대안이 아니다.
