# BIZ-003 서울 돌잔치 비교 필드·행사 분류·신뢰 라벨 정책안

- 작업 ID: `BIZ-003`
- 작성 기준일: 2026-07-22 (Asia/Seoul)
- 문서 상태: **정책안 작성 완료 / 최종 공개 문구·재확인 주기는 D-25 APPROVAL_REQUIRED**
- 적용 대상: 전국 가족행사 플랫폼 중 첫 검증 코호트인 `서울 돌잔치`
- 수정 범위: 이 보고서 1개 신규 작성
- 비변경 범위: 제품·데이터·DB/API·라우팅·디자인 토큰·환경변수·패키지·CHG-A~C·외부 실행

## 1. 결론과 정책 지위

서울 돌잔치 비교의 기본 단위는 `업체 전체 신뢰 점수`가 아니라 **같은 카테고리의 필드별 조건**이다. 장소와 관련 서비스는 서로 비교하지 않고, 가격·인원·주차·시설은 단위·포함 범위·조건·확인일이 같은 경우에만 나란히 놓는다. 결측값은 0 또는 `미제공`으로 추정하지 않는다.

신뢰 표시는 다음 네 축을 합치지 않는다.

1. 출처: 공공데이터 또는 승인된 업체 담당자 중 누가 값을 제공했는가
2. 사업자·소유권: 제출된 사업자 상태와 페이지 편집 관계를 어디까지 확인했는가
3. 필드 검수: 관리자가 어느 값의 출처·조건·권리를 검토했는가
4. 최근성: 해당 값을 언제 확인했고 언제 다시 확인해야 하는가

공개 라벨은 `공공데이터 등록`, `정보 확인 중`, `업체 직접 제출`, `사업자 확인`, `관리자 검수`, `최근 정보 확인 필요` 여섯 개를 사용한다. 라벨 여러 개가 동시에 적용될 수 있으며, 어느 하나도 다른 축을 대신하지 않는다.

이 정책안은 다음 선행 결정을 그대로 따른다.

- **확정:** 전국 가족행사 범위를 유지하고 서울 돌잔치는 초기 운영 우선순위로만 둔다(`ADR-001`, `ADR-016`).
- **확정:** 1차 전환 종점은 비교·견적 문의·업체 응답까지다. 예약·결제·에스크로·정산은 보류한다(`ADR-011`, `ADR-016`).
- **확정:** NAVER 파생 4,960건은 `legacy_source_hold`이며 공개 재고·SEO 수치·서울 파일럿 분모에서 제외한다(`ADR-016`).
- **확정:** 후기 수는 공개 자격이 아니며 `api_collected`는 검증이 아니다. 결측값은 `정보 없음/확인 필요`로 처리한다(D-01~D-03).
- **제안:** 아래 카테고리별 비교 최소 자격, 화면 설명문, 90/120/180/365일 주기는 D-25 승인 전 내부 계약이다.
- **승인 필요:** 실제 공공데이터 원천은 D-23, 개인정보·증빙은 D-24, 최종 라벨·주기는 D-25, 업체 연락·파일럿 실행과 통과선은 D-26을 따른다.

## 2. 기존 상태와의 연결

새로운 업체 lifecycle을 만들지 않고 BIZ-002·BE-005 상태를 다음처럼 사용한다.

| 사업 단계 | 기존 상태 | BIZ-003에서의 의미 | 공개·행동 |
| --- | --- | --- | --- |
| 레거시 격리 | T0 `legacy_source_hold` | NAVER 파생 후보 | 공개·집계·비교·문의 모두 금지 |
| 공공 후보 | T1 `public_candidate` | 허용 원천에서 기본 사실 후보 발견 | 기본적으로 내부 검수 큐 |
| 정체성 정리 | T2 `identity_checked` | 실체·지점·주소·영업 상태 후보 정리 | 행사 가능을 뜻하지 않음 |
| 행사 후보 | T3 `event_fit_candidate` | 질문표상 돌잔치 제공 가능성이 있음 | 업체 확인 또는 운영 검수 전 확정 표시 금지 |
| 관계 확인 | T4 `owner_business_checked` | 사업자 상태와 담당자 관계를 각각 확인 | 공개·문의가 자동으로 열리지 않음 |
| 필드 검수 | T5 `field_reviewed` | 승인된 필드만 공개 projection에 반영 | 기본 페이지·조건 충족 시 비교 가능 |
| 문의 준비 | T6 `inquiry_ready` | 문의 opt-in·업무 연락·제품 E2E까지 확인 | 견적 문의 가능 |

`정보 확인 중`은 위 단계를 대체하는 새 상태가 아니다. 신규 후보의 미검수 상태, 기존 공개 필드의 변경·분쟁·재확인 상태를 사용자에게 설명하는 표시다.

## 3. 서울 돌잔치 장소·서비스 분류

### 3.1 분류 원칙

```text
독립 출처 후보
→ 동일 실체·지점·주소·영업 adverse signal 확인
→ 서울 소재 장소 또는 서울 제공 서비스인지 확인
→ 돌잔치 제공 근거 확인
→ 장소/서비스와 세부 비교군 분류
→ 업체 주장 또는 운영 확인을 관리자 검수
→ 기본 페이지 → 같은 비교군 비교 → T6만 견적 문의
```

- 공공 인허가·상권 업종은 후보 신호일 뿐 `돌잔치 가능`의 증거가 아니다.
- 장소는 실제 행사 위치, 서비스는 제공 지역을 기준으로 서울 파일럿 포함 여부를 판단한다.
- 같은 브랜드라도 주소·전화·인허가가 다른 지점은 별도 업체다.
- 장소와 서비스, 서로 다른 서비스군을 하나의 가격 순위로 섞지 않는다.
- 한 업체가 장소와 서비스를 함께 제공하면 capability와 가격 묶음을 각각 기록하되, 패키지의 포함·제외를 명시한다.
- 기존 8개 행사 분류와 URL은 유지한다. 서울 파일럿의 `돌잔치`는 기존 `kids(돌잔치·백일)` 안의 조건부 capability로 구분하는 사업 개념이며, 새 행사 유형·API 필드·라우팅을 이 보고서에서 만들지 않는다. 실제 매핑은 BE-006에서 충돌 여부를 검토한다.

### 3.2 비교군 사전

| 대분류 | 비교군 | 포함 정의 | 비교 핵심 |
| --- | --- | --- | --- |

| 서비스 | 의상·뷰티 | 돌잔치용 의상 대여 또는 헤어·메이크업을 서울 고객·행사에 제공한다고 확인된 업체 | 대상 인원, 피팅·출장, 대여 기간, 보증금·추가비 |

| 서비스 | 기타 돌잔치 서비스 | 사회·공연 등 돌잔치에 직접 제공되는 서비스로 범위와 결과물이 확인된 업체 | 서비스 단위, 제공 시간, 인력·장비, 출장·취소 조건 |

`기타`는 비교 편의를 위한 무제한 수용함이 아니다. 표준 단위가 없는 업체는 개별 기본 페이지까지만 허용하고, 동일 비교군이 정의되기 전 가격 순위·비교표에 넣지 않는다.

### 3.4 포함·제외 판정

**포함**

- 서울 소재 장소가 돌잔치 제공을 명시하고 해당 capability가 관리자 검수됨
- 서울 제공 범위와 출장·배송 조건이 있는 돌잔치 관련 서비스가 검수됨
- 공공데이터에서 독립 재발견한 업체라도 행사 capability는 업체 주장 또는 운영 근거로 별도 검수됨
- 가격이 없어도 기본 공개 gate를 만족하면 페이지 공개 가능. 비교 자격은 7절을 추가 적용

**제외**

- NAVER 파생값만 있는 `legacy_source_hold`
- 일반 호텔·음식점·공간·사진관이라는 업종만 있고 돌잔치 제공 근거가 없는 후보
- 서울 밖 장소를 서울 장소처럼 표시하는 경우
- 돌잔치와 직접 관련 없는 소매·커뮤니티·콘텐츠 작성자·대관 중개 목록
- 폐업 확정, 범위 밖, 권리 차단, 강한 중복 미해결, 관리자 publication 반려

**확인 필요**

- 돌잔치 제공 여부, 지점/주소, 서비스 지역, 가격 포함 범위, 좌석 인원, 주차 배정, 외부업체 반입, 사진 권리 중 하나라도 근거가 불명확한 경우
- 확인 필요 후보는 내부 큐에서 유지한다. 기존 공개 페이지의 일부 필드가 확인 중인 경우에만 이전 승인값과 상태를 함께 표시할 수 있다.

## 4. 비교 필드 공통 계약

이 사전은 제품 스키마가 아니라 BE-006이 데이터 계약으로 매핑할 **사업 의미 사전**이다. 필드명은 논리명이며 실제 DB 이름을 확정하지 않는다.

### 4.1 모든 공개 비교값의 공통 메타데이터

| 논리 필드 | 의미 | 필수 규칙 |
| --- | --- | --- |
| `comparison_group` | 동일 비교군 | 같은 그룹끼리만 가격·조건 비교 |
| `event_type` | `kids` 중 돌잔치 capability | 백일·아동 행사 일반과 돌잔치 근거를 혼합하지 않음 |
| `condition_scope` | 요일·시간대·행사 월·룸·패키지·지점 등 값의 적용 조건 | 조건이 다르면 동일값처럼 정렬 금지 |
| `source_layer` | public_data / provider_supplied / operator_verified / approved_user_case | 사용자 제보 사례는 현재 업체값과 분리 |
| `source_reference` | dataset/record 또는 비공개 증거와 연결되는 재현 ID | 공개 화면에는 허용된 출처명·링크만 표시 |
| `basis_date` | 원천 기준일 또는 업체가 조건을 확인한 날짜 | 수집일·기술 `updated_at`으로 대체 금지 |
| `submitted_at` | 업체·이용자가 값을 제출한 날짜 | 사실 확인일이 아니라 제공일로 표시 |
| `verified_at` | 관리자가 해당 필드와 근거를 검수한 날짜 | provider 전체 확인일로 확장 금지 |
| `valid_until` | 업체가 명시한 금액·조건의 유효 종료일 | 재확인 기본 주기보다 우선 |
| `next_check_at` | 정책상 다음 확인 예정일 | 공개 확인일과 구분 |
| `review_status` | pending/approved/rejected/disputed/expired/revoked | approved이고 유효한 assertion만 현재값 후보 |
| `missing_reason` | unknown/not_provided/not_applicable/withheld/pending_review | null 하나로 의미를 합치지 않음 |

### 4.2 결측·0·해당 없음

| 값 | 화면 표현 | 필터·비교 동작 |
| --- | --- | --- |
| `unknown` | `정보 없음` | 조건 충족으로 간주하지 않음 |
| `not_provided` | `업체 미제공` | 0·없음으로 간주하지 않음 |
| `not_applicable` | `해당 없음` | 분모에서 제외, 0점 처리 금지 |
| `withheld` | `공개하지 않은 정보` | 내부 이유·권한은 공개하지 않음 |
| `pending_review` | `정보 확인 중` | 새 값은 정렬·필터에 사용하지 않음 |
| 확인된 숫자 `0` | `0원`, `0대`처럼 단위 포함 | 근거·조건이 승인된 경우에만 실제 0으로 사용 |

`미제공`을 경쟁상 불리한 값으로 점수화하지 않고, `정보 없음` 업체를 저가·무료·주차 불가로 정렬하지 않는다.

## 5. 가격·인원·주차·시설 비교 사전

### 5.1 가격

모든 금액은 원화 `KRW`, 정수 원 단위로 저장·표시한다. 부가세·봉사료·배송·출장·설치비를 포함한 총액인지 분리하지 못하면 `총액`으로 비교하지 않는다.

| 논리 필드 | 단위/값 | 적용 조건 | 결측·만료 처리 |
| --- | --- | --- | --- |
| `pricing_model` | per_person / package / rental / per_item / hourly / quote_only / hybrid | 비교군별 허용 모델 | 없으면 금액 비교 불가 |
| `adult_meal_price` | 원/성인 1인 | 메뉴·요일·시간대·세금·봉사료 조건 | 0 추정 금지 |
| `child_meal_price` | 원/아동 1인 | 연령 기준과 메뉴 범위 | 성인가의 비율로 추정 금지 |
| `venue_rental_fee` | 원/회 | 기본 이용 시간·룸·요일 | 무료라고 확인된 경우만 0원 |
| `package_price_min/max` | 원/패키지 | 패키지명·인원·결과물·포함 품목 | 최솟값만 떼어 대표 가격으로 사용 금지 |
| `item_unit_price` | 원/개 또는 원/세트 | 답례품·초대장 등 품목과 최소 수량 | 최소 주문과 함께 표시 |
| `hourly_price` | 원/시간 | 기본/추가 시간, 인력·장비 범위 | 다른 패키지 총액과 직접 순위 금지 |
| `minimum_order_amount` | 원/주문 | 케이터링·상품의 최소 주문 | 패키지 가격과 구분 |
| `travel_delivery_fee` | 원/건 또는 산정 조건 | 서울 구·거리·주차·배송 조건 | 미정이면 총액 계산 금지 |
| `deposit_or_guarantee` | 원 또는 비율 | 예약금·보증금의 성격과 반환 조건 | 결제 기능으로 오인시키지 않음 |
| `tax_included` | yes/no/unknown | 부가세 포함 여부 | unknown이면 총액 확정 표현 금지 |
| `service_charge` | 원 또는 비율 | 봉사료 적용 대상 | 금액에 합산 여부 명시 |
| `included_items[]` | 표준 항목 목록 | 돌상·장식·촬영·식사·음료·인력 등 | 빈 배열을 `포함 없음`으로 해석 금지 |
| `excluded_items[]` | 표준 항목 목록 | 별도 비용 항목 | 공개값과 같은 조건일 때 필수 |
| `valid_until` | YYYY-MM-DD | 업체 명시 유효기간 | 경과 다음 날 금액 숨김 |

가격 비교 규칙:

1. `per_person`, `package`, `rental`, `per_item`, `hourly`는 같은 열 또는 한 순위에 섞지 않는다.
2. 추정 총액은 사용자가 인원·선택 항목을 입력하고 모든 구성값의 포함 범위가 맞을 때만 `예상 합계`로 계산하며 `실견적 아님`을 표시한다.
3. `quote_only`는 가격 누락이 아니라 `조건 입력 후 견적`이라는 모델이다. 다만 T6가 아니면 견적 CTA를 열지 않는다.
4. 이용자 제보 견적은 행사 시점·인원·포함/제외가 완전한 `과거 사례`로만 표시하고 업체 현재 가격 열에 합치지 않는다.
5. `최저가`, `평균가`, `실시간 가격`, `예약가`는 모집단·계산·실시간 계약이 없으므로 사용하지 않는다.

### 5.2 인원

| 논리 필드 | 단위 | 의미와 조건 | 금지 해석 |
| --- | --- | --- | --- |
| `minimum_bookable_guests` | 명 | 예약 가능한 최소 인원 | 보증 인원과 자동 동일시 금지 |
| `minimum_guaranteed_guests` | 명 | 최소 결제 또는 보증 기준 인원 | 실제 참석 최소 인원으로 표현 금지 |
| `maximum_seated_guests` | 명 | 해당 룸·배치의 좌석 수용 상한 | 장소 전체 정원과 혼합 금지 |
| `maximum_standing_guests` | 명 | 입석을 별도 제공할 때만 | 돌잔치 기본 수용 인원으로 사용 금지 |
| `included_guest_count` | 명 | 패키지 가격에 포함된 기준 인원 | 최대 수용 인원으로 사용 금지 |
| `extra_guest_unit_price` | 원/명 | 포함 인원 초과 시 추가 금액 | 성인·아동 구분 없으면 단일값 확정 금지 |
| `guest_age_rule` | 텍스트/구간 | 성인·아동·유아 가격의 연령 기준 | 손품해방 임의 연령 기준 사용 금지 |
| `capacity_condition` | 룸·좌석 배치·돌상·무대 조건 | 인원 수치가 성립하는 배치 | 조건 없는 최대치 강조 금지 |

인원은 항상 `명`으로 표시하고, 아동·유아를 총인원에 포함하는지 조건을 붙인다. 장소 비교 진입에는 관리자 검수된 `maximum_seated_guests` 또는 명확한 좌석 범위가 필요하다.

| `free_parking_vehicles` | 대 | 무료 등록 가능한 차량 수 | 무료 시간과 함께 표시 |

### 5.4 시설·이용 조건

시설은 `yes/no/unknown/not_applicable`의 4상태와 사용 조건을 함께 가진다. `yes`는 실제 이용 가능 범위가 확인된 경우에만 사용한다.

| 필드군 | 표준 항목 | 필수 조건 |
| --- | --- | --- |
| 공간 | 독립 룸, 단독 대관, 돌상 설치 공간, 대기·수유 공간 | 독립 범위, 동시간 타 행사, 예약 조건 |
| 영유아 | 유아 의자, 기저귀 교환대, 수유 공간, 유모차 이동 | 수량·위치 또는 사전 요청 여부 |
| 이동·접근 | 엘리베이터, 계단 없는 경로, 휠체어 접근 경로, 장애인 화장실 | 특정 동선 확인이며 시설 전체의 법적 적합 인증 아님 |
| 진행 장비 | 마이크, 음향, 프로젝터/화면, 전원, 테이블·의자 | 무료/유료, 운영 인력, 반입 가능 여부 |
| 음식·외부업체 | 외부 음식, 케이크, 돌상, 촬영, 사회자·공연 반입 | 허용/금지/협의와 반입비·지정업체 조건 |
| 시간·운영 | 기본 이용 시간, 준비·철거 시간, 연장 단위, 동시 행사 | 가격 필드와 같은 요일·룸 조건 |

`가족 친화`, `배리어프리`, `아기 편의 완비`처럼 여러 필드를 하나로 묶는 표현은 전 항목의 정의와 근거가 승인되기 전 사용하지 않는다.

## 6. 최근성·유효기간 정책안

아래 숫자는 BE-005의 90/120/180일 기준을 구체화한 **D-25 승인 전 제안값**이다. `valid_until`이나 권리 철회·분쟁·adverse signal이 더 이르면 즉시 그 조건을 우선한다.

### 6.1 날짜 기준

- 업체가 자사 값을 실제로 재확인한 경우 `provider_attested_at`을 값의 확인 기준일로 쓴다.
- 운영자가 독립 공식 근거로 사실을 재확인한 경우 해당 필드의 `verified_at`을 기준일로 쓸 수 있다.
- 단순 관리자 형식 검수, 행 `updated_at`, 수집 실행일은 값의 최근성을 갱신하지 않는다.
- 여러 근거가 있으면 공개 projection이 선택한 assertion의 기준일을 사용한다.
- 일수는 기준일 다음 날을 1일로 계산하는 KST 달력일 기준 제안이다. 구현 계약은 BE-006에서 고정한다.

### 6.2 제안 주기와 화면 동작

| 대상 | 0~90일 | 91~120일 | 121~180일 | 181일 이상 |
| --- | --- | --- | --- | --- |
| 가격·취소·추가비 | 현재값 표시 가능 | `최근 정보 확인 필요`, 가격 정렬·최저 표현 금지 | 금액 숨김, `가격 정보 재확인 중` | 금액 숨김 유지 |
| 좌석 인원·주차·시설·행사 capability | 현재값 표시 가능 | 라벨 표시, 재확인 큐 | 라벨과 마지막 확인일을 표시하되 필터의 확정 충족값으로 사용하지 않음 | 비교값을 `정보 확인 필요`로 대체 |
| 업체명·주소·업무 연락 | 현재값 표시 가능 | 재확인 큐 | 충돌 신호가 없으면 경고 표시 | 기본 페이지 `needs_update` 검토 |
| 사업자 상태 | current | 재조회 필요 | 문의 보류 검토 | 문의 금지, 긴급 검수 |
| 문의 준비 핵심 정보 | 가능 | 경고·재확인 | 180일까지 운영 판단 | 문의 자동 중지 |

추가 규칙:

- 업체가 명시한 `valid_until`이 기준일+90일보다 빠르면 만료 다음 날부터 가격을 숨긴다.
- 권리 조건 변경, 사업자 휴·폐업 신호, 주소 충돌, 업체 이의제기, 관리자 반려는 일수와 무관하게 `정보 확인 중` 또는 `최근 정보 확인 필요`를 적용한다.
- ownership 관계 365일 재확인은 BE-005의 운영 제안값으로 유지하되, D-24·OPS-008과 함께 승인한다. 365일이 지나면 grant 재확인 대상이며 사업자 상태와 합치지 않는다.
- 오래된 가격을 0원 또는 `문의 시 확정`으로 자동 바꾸지 않는다.

## 7. 6개 공개 신뢰 라벨

### 7.2 적용 조건·화면 문구·금지 의미

| 라벨 | 적용 조건 | 화면 보조 문구안 | 이 라벨이 뜻하지 않는 것 |
| --- | --- | --- | --- |

| `사업자 확인` | 업체가 제출한 사업자번호의 진위/상태 결과가 current이고 provider 연결을 관리자가 검토 | `제출된 사업자등록정보의 상태를 확인했어요. 담당자 권한이나 가격·시설 전체를 인증한 것은 아닙니다.` | 소유권 자동 승인, 안전·품질 인증, 가격·사진·행사 가능 검증 |
| `관리자 검수` | 해당 공개 assertion의 출처·단위·조건·권리·결측·정책 적합성을 reviewer가 승인 | `손품해방 운영자가 이 항목의 출처와 표시 조건을 검수했어요.` | 공공기관 인증, 계약 보증, 미래 가격·재고 보장, 업체 전체 검증 |
| `최근 정보 확인 필요` | 제안 주기 초과, `valid_until` 만료, 충돌·이의·원천 변경 또는 재확인 실패 | `마지막 확인 후 시간이 지났거나 변경 신호가 있어요. 문의 전 업체에 다시 확인해 주세요.` | 폐업 확정, 정보 오류 확정, 자동 삭제 |

### 7.3 조합 규칙

- `공공데이터 등록 + 관리자 검수`: 공공 필드의 출처·표시 조건을 관리자가 검수했다는 뜻이다. 업체 참여를 뜻하지 않는다.
- `업체 직접 제출 + 사업자 확인 + 관리자 검수`: 제출 주체, 사업자 상태, 해당 필드 검수가 각각 완료됐다는 뜻이다. 가격 유효기간과 문의 준비는 별도다.
- 위 세 라벨에 `최근 정보 확인 필요`가 붙으면 오래된 값이 우선한다. 가격은 숨기고 나머지 필드는 필터 확정값에서 제외한다.
- `정보 확인 중`인 새 값은 비교·정렬에 쓰지 않는다. 기존 승인값이 아직 유효하면 기존값과 `변경 정보 확인 중`을 함께 표시하고, 기존값도 만료됐으면 값을 숨긴다.
- 라벨은 업체별 단일 대표 배지로 압축하지 않는다. 카드 요약에는 출처 1개와 가장 보수적인 최근성만 보여도 상세·비교에는 필드별 근거와 날짜를 열어야 한다.

### 7.4 금지 문구

근거 계약과 별도 승인이 없는 다음 표현은 0건이어야 한다.

- `공식 인증`, `공식 인증 업체`, `검증 완료 업체`, `100% 검증`, `믿을 수 있는 업체`
- `최저가`, `실시간 가격`, `예약 가능`, `즉시 예약`, `예약 확정`, `결제 가능`
- `추천`, `인기`, `후기 많은`, `평점 좋은`, `서울 1위`, `베스트`
- `사업자 인증 완료`, `소유권 인증`을 업체 전체 정보 정확성의 의미로 쓰는 표현
- `무료`, `주차 가능`, `장애인 이용 가능`을 결측·부분 조건에서 확정하는 표현
- 미확정 Premium 가격·리드 과금·광고 혜택·중개 수수료

## 8. 화면·행동별 최소 자격

### 8.1 기본 업체 페이지

BE-005의 기본 공개 10개 gate를 모두 만족해야 한다.

1. stable provider ID와 canonical match, 미해결 강한 중복 없음
2. 업체명·표준 업종·서울 주소 또는 서울 서비스 지역이 승인된 assertion으로 존재
3. 승인된 공공데이터 또는 업체 공식 채널 등 재현 가능한 출처가 있고, 적용되는 권리·출처표시 검토가 유효
4. 폐업·휴업·이전의 미해결 authoritative adverse signal 없음
5. 돌잔치 capability가 관리자 검수된 `provider_asserted` 또는 `operator_confirmed`
6. 최소 식별 필드가 만료되지 않고 수집일·기준일·확인일이 분리됨
7. 관리자 publication review와 결정 사유·정책 버전·reviewer 기록
8. 공개 projection에 사업자번호·개인정보·비공개 증빙·권리 미확인 사진 없음
9. 공개값의 출처 라벨과 최근 확인일을 표시 가능
10. 결측을 0·false·무료로 추정하지 않음

공공데이터 후보 또는 `정보 확인 중`인 신규 업체는 위 gate 전 공개 기본 페이지를 만들지 않는다. 기존 승인 페이지의 일부 필드가 확인 중인 경우만 기존값과 상태를 보존한다.

### 8.2 비교 진입

기본 페이지 자격에 더해 다음을 만족해야 한다.

**공통**

- 같은 `comparison_group`과 같은 돌잔치 capability끼리 비교
- 모든 비교 필드가 승인값 또는 명시적 결측 상태를 가짐
- 필드별 제공 주체·기준일/확인일·검수·최근성 표시 가능
- expired·disputed·pending 새 값을 정렬·필터의 확정값으로 사용하지 않음
- 현재 기존 비교 계약인 최대 3곳을 유지

**장소**

- 좌석 기준 최대 인원 또는 검수된 좌석 범위가 필수
- 가격은 `quote_only` 또는 `정보 없음`일 수 있으나 금액이 있으면 가격 사전을 완전 충족
- 주차와 주요 시설은 yes/no를 강제하지 않고 각 상태와 조건을 명시

**서비스**

- 서울 제공 지역, 결과물/서비스 단위, 제공 조건이 필수
- 가격 모델은 금액형 또는 `quote_only` 중 하나를 명시
- 장소에만 해당하는 인원·주차·시설은 `not_applicable`로 처리하고 감점하지 않음

비교 자격은 추천 자격이 아니다. 기본 자격을 만족해도 카테고리 필수 단위가 없으면 상세 페이지만 제공하고 `비교 정보 준비 중`으로 둔다.

### 8.3 견적 문의 진입

비교 자격과 BE-005의 T6 조건을 모두 만족해야 한다.

1. ownership claim 승인과 active access grant
2. 제출 사업자번호 상태가 current active이며 invalid/closed 신호 없음
3. 업체의 명시적 문의 수신 opt-in
4. 승인된 업무 연락 경로와 inquiry responder
5. 핵심 정보가 180일 이내이며 반복 미응답 제한 통과
6. RLS·문의 생성·선택 업체 전달·열람·응답·알림 실패 E2E 통과

T6가 아니면 `견적 문의 가능`, `예약 가능`으로 표시하지 않는다. 문의 후 계약은 사이트 밖에서 진행하며 손품해방 예약·결제 완료처럼 표시하지 않는다.

### 11.3 파일럿 게이트

**즉시 중단/보완**

- NAVER 파생값 재승계, 사업자번호·증빙 공개, 근거 없는 사진, 미표시 광고, 미확정 가격·수수료·예약 기능 공개가 1건이라도 발생
- 기본 페이지·비교·문의 자격을 건너뛴 공개 또는 선택 업체 외 문의 전달 발생
- 공공데이터 등록·사업자 확인·관리자 검수를 하나의 `검증 완료`로 축약

**다음 단계 판단을 위한 기존 제안값**

- T6 문의 가능 업체 20곳 이상, 단 한 유형·한 구 편중이면 미통과
- 전달 성공 유효 문의 30건 이상. 미달이면 전환율 결론을 내리지 않음
- 전달 성공률 95% 이상, 성공으로 오표시된 실패 0건
- 유효 문의의 업체 응답률 50% 이상, 첫 응답시간 중앙값 측정 가능

위 네 수치는 BIZ-002에서 가져온 승인 전 내부 제안값이다. BIZ-003이 새 성과 약속으로 확정하지 않으며 D-26에서 코호트·운영량과 함께 결정한다.
