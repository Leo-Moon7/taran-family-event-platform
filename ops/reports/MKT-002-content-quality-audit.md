## 3. 감사 방법과 판정 기준

### 3.1 재현 방법

- 번들 Node로 `blog-data.js`의 `window.taran_BLOG_POSTS` 배열을 읽기 전용 파싱
- 글별 제목, slug, category, date, readTime, sections, body, checklist, questions, pitfalls, nextActions, summaryPoints, tags 집계
- 문단 완전 일치 빈도와 문장 빈도를 계산하고, 5회 이상 나온 문장을 반복 문장군으로 분류
- 글별 `반복 문장 문자 수 ÷ 전체 본문 문자 수`로 템플릿 비율 계산
- `blog.js`의 CTA·관련 글 선택 로직을 동일한 배열 순서로 재현
- `article-contract-questions.html`의 H1·H2·본문·목록을 쿼리 글과 항목별 대조
- `rg`로 가격·비율·기간·예약·결제·에스크로·정산·검수·포인트 표현과 정적 링크를 교차 확인

### 3.2 얇은 콘텐츠 판정

글자 수 하나로 판정하지 않고 다음을 함께 봤다.

1. 반복 문장 비율
2. 제목과 본문 산출물의 일치 여부
3. 주제별 고유 설명·사례·표·공식·체크 방법의 존재 여부
4. 가격·기간·규정·기능 주장의 근거와 기준일
5. 같은 검색 의도를 가진 다른 글 또는 정적 페이지의 존재
6. 행사·준비 단계·사용자 행동과 연결되는 문맥형 내부 링크

## 11. 내부 링크 공백과 권고

### 현재 사실

- 28개 데이터 객체에 본문 링크 또는 관련 대상 필드가 없다.
- `blog.js`는 글의 카테고리·제목·태그에 특정 단어가 있으면 계산기·체크리스트·업체 목록 중 하나를 자동 선택한다.
- CTA 분포는 업체 목록 20, 체크리스트 6, 계산기 2다.
- 관련 글은 같은 카테고리를 우선한 뒤 배열 앞쪽 글로 채운다. 카테고리가 대부분 1개 글만 갖기 때문에 21개 글이 동일한 앞 3개 글로 연결된다.
- 글 안에서 편집 정책, 운영 정책, 개인정보처리방침, 문의, 행사별 허브, 관련 업체 조건으로 이어지는 문맥 링크는 없다.

## 13. URL 결정 필요 항목

이번 작업에서 URL을 확정하지 않았다. 총괄 PM과 D-10에서 다음을 결정해야 한다.

1. `article.html?slug=contract-questions`와 `article-contract-questions.html` 중 대표 경로

3. `guides.html`을 요약 허브로 유지할지, 준비백과 글과 통합할지
