## 1. 비개발자용 한 문장 결론

**업체명·주소 같은 공개 사실을 다른 공식 출처에서 직접 확인해 손품해방의 정보로 만드는 것은 가능하지만, NAVER API가 돌려준 수천 건을 그대로 저장·분석해 자체 업체 DB와 추천 정보로 공개하는 것은 별도의 이용 허락을 확인하기 전에는 허용된다고 볼 수 없다.**

여기서 중지 대상으로 판단한 것은 `모든 업체 정보`나 `사이트 전체`가 아니다. 현재 NAVER 검색 결과를 원천으로 만든 정적 데이터와 그 파생 수치·분류·추천의 공개를 말한다. 원본은 바로 삭제하지 않고 비공개로 보존하면서, 업체 공식 사이트·허용된 공공데이터·업체 직접 입력·사용자 제보를 통해 같은 업체의 기본 사실을 독립적으로 다시 확인할 수 있다.

## 2. QA-007에서 정교하게 바로잡을 부분

QA-007의 최종 안전 결론은 유지되지만, 계약 근거는 다음처럼 더 정확히 구분해야 한다.

1. 현재 수집 코드는 `naverapihub.apigw.ntruss.com`을 사용하므로 우선 확인할 계약은 NAVER Cloud의 현행 `AI·Naver API 서비스 이용약관`, 콘솔 신청 시 동의한 조건, 등록 애플리케이션의 목적·권한이다.
2. 현행 공개 약관 제5조 제8항은 회사의 사전 동의 없이 결과 데이터를 허용 범위를 넘어 복제·저장·가공·배포하거나 제3자에게 제공하지 못하게 한다.
3. 현행 검색 API 기술 문서는 검색 결과 조회 방법과 필드를 설명하지만, 지역 결과 수천 건의 장기 저장·자체 DB 공개나 블로그 결과의 추천 점수화를 허용한다고 명시하지 않는다. API가 필드를 반환한다는 사실과 그 필드를 영구 저장·재배포할 권리는 다르다.
4. NAVER Developers의 기존 검색 API 약관은 지역정보를 별도 DB로 관리하는 행위를 금지 예시로 들고 검색 결과를 독립적으로 표시하도록 한다. 다만 API HUB에 이 기존 약관이 그대로 계약 내용으로 편입됐는지는 실제 콘솔 계약 기록 없이는 확정할 수 없으므로, QA-009는 이를 단독 법적 근거가 아니라 강한 보충 근거로 사용한다.
5. 따라서 현재 흐름을 곧바로 `불법 확정`이라고 말할 수는 없지만, 공개 자료만으로 `허용 확정`이라고 말할 수도 없다. 현재 보유 증거상 공개 확대를 중지하고 NAVER의 사전 동의 또는 실제 계약 범위를 확인하는 것이 맞다.

## 4. 왜 업체 기본 사실은 사용할 수 있고 현재 DB 방식은 멈춰야 하는가

### 4.1 업체명·주소 같은 사실

업체명, 업종, 영업장 주소, 대표전화, 공식 홈페이지 주소는 일반적으로 짧은 사실 정보다. 저작권법은 창작적 `표현`을 보호하고, 데이터베이스 권리가 개별 사실 그 자체에 미치는 것은 아니라고 규정한다. 따라서 손품해방이 아래 출처에서 사실을 독립적으로 확인해 자체 필드로 기록하는 방식은 가능성이 높다.

- 업체가 직접 제공하고 공개에 동의한 정보
- 업체가 운영하는 공식 홈페이지·공식 예약 채널에서 수동 확인한 최소 사실
- 명시된 이용허락 범위 안의 공공데이터
- 이용자가 제보하고 운영자가 다른 출처로 재확인한 정보

다만 `NAVER에 공개되어 있으니 NAVER 결과를 그대로 복사해도 된다`는 결론은 나오지 않는다. 사실 자체에 저작권이 없더라도 그 사실을 얻은 API의 계약, 데이터베이스제작자 권리, 개인정보, 상표·오인, 정확성 책임이 따로 남는다.

### 4.2 검색 결과의 집합과 가공물

NAVER가 구성해 반환한 지역 결과 5,031건과 블로그 검색 결과 9,991건은 단일 업체 사실 하나가 아니라 검색·정렬·선택된 결과의 집합이다. 이를 반복 호출해 정적 파일이나 SQLite에 장기 저장하고, 다시 업체·지역·행사·가격·인원·후기 수로 가공해 공개하면 다음 행위가 동시에 발생한다.

- API 결과의 대량 복제와 저장
- 별도 업체 DB 구축
- 검색 결과를 자체 정보로 가공
- 제3자에게 정적 파일로 배포
- 원래 검색 맥락을 제거하고 추천·검증 정보처럼 표시

이 단계는 `공개된 사실 한 건을 독립 확인`하는 것과 다르다. 현행 NCP 약관 제5조 제8항의 사전 동의·허용 범위 확인 대상이고, 저작권법 제93조 제2항의 반복적·체계적 복제 위험도 생긴다.

### 4.3 블로그 후기

블로그의 URL·작성일은 사실 성격이 강하지만 제목·문장·사진·후기 구성은 작성자의 표현일 수 있다. 현재 코드는 본문·사진·작성자 프로필을 복제하지 않는 점은 좋다. 그러나 검색 API의 제목·요약·URL·날짜를 장기 저장한 뒤 요약문에서 가격·인원·행사 유형을 추출하고 `후기 기반 등록`, `원문 수`, 추천 근거로 바꾸는 것은 저작권 문제와 별도로 API 결과 `저장·가공` 문제에 해당한다.

원문 링크만 건다고 모든 문제가 해결되는 것도 아니다. 링크 출처가 이용자 제보인지 API 결과인지, 링크 제목을 복제하는지, 삭제된 원문에서 만든 태그·수치가 계속 남는지를 관리해야 한다.

### 6.3 저작권·개인정보·표시 문제의 분리

- **저작권:** 업체 기본 사실 자체는 보호 대상이 아닐 가능성이 크지만 블로그 문장·사진은 별도다.
- **데이터베이스권:** 개별 사실이 아니라 NAVER가 만든 대량 결과 집합을 반복적·체계적으로 복제하는 문제다.
- **API 계약:** 저작권 침해가 아니어도 API 결과의 저장·가공·배포를 계약이 제한할 수 있다. 현재 흐름의 가장 직접적인 미확정 사항이다.
- **개인정보:** 회사 대표번호는 보통 회사 정보지만, 개인사업자 실명·휴대전화·자택 주소는 개인을 식별할 수 있어 별도 처리 근거·최소화가 필요하다.
- **표시·정확성:** `NAVER에 검색됨`을 `손품해방 확인`, `후기 많음`, `평점 좋음`, `추천 업체`로 바꾸면 소비자 오인이 생길 수 있다.

## 7. 안전하게 업체 정보를 채우는 최소 방식

사용자가 원하는 `입점 업체가 없어도 유용한 정보 플랫폼` 방향은 유지할 수 있다. NAVER 결과에 의존하지 않는 공개 레코드를 다음 순서로 만든다.

1. **후보 발견:** NAVER·공공데이터·검색엔진·사용자 제보 등으로 후보를 발견하되, NAVER API 결과의 장기 보관 여부는 계약 확인 전 중단한다.
2. **독립 재확인:** 업체 공식 홈페이지, 허용된 공공데이터, 업체 직접 제출 중 하나 이상에서 업체명·업종·영업장 주소·대표전화·공식 URL을 다시 확인한다.
3. **출처별 필드 저장:** `값`, `출처 유형`, `출처 URL`, `확인일`, `확인자`, `개인정보 여부`, `다음 확인일`을 함께 둔다. NAVER에서 본 값을 그대로 옮긴 뒤 출처만 공식 홈페이지로 바꾸면 안 된다.
4. **최소 공개:** 초기에는 업체명·업종·지역·공식 URL·확인일만 공개한다. 가격·수용 인원·평점·후기 수는 업체 제공, 조건 있는 사용자 제보 또는 운영 확인 전까지 `정보 없음`으로 둔다.
5. **라벨 분리:** `업체 제공 정보`, `공공데이터`, `운영팀 확인`, `이용자 제보`를 구분한다. `NAVER 검증`, `공개 후기 기반 확인`, `추천` 표현은 사용하지 않는다.
6. **정정·삭제:** 업체·작성자·이용자가 대상 URL과 사유를 제출할 수 있게 하고, 원천 삭제 시 공개 필드와 파생 태그·수치까지 함께 회수한다.
7. **보관 기간:** 후보·출처별 보관기간과 재확인일을 정하고 기한 만료 시 자동 공개 유지가 아니라 재검수로 보낸다.

### 권장 공개 레코드 예시

```text
업체명: OO연회장
업종: 연회장
지역: 서울 송파구
공식 홈페이지: https://example.com
정보 출처: 업체 공식 홈페이지
운영팀 확인일: 2026-07-22
가격·수용 인원: 정보 없음
수정 요청: 접수 가능
```

이 방식에서는 같은 업체를 처음 NAVER에서 발견했더라도 최종 공개 사실은 별도의 허용된 출처에서 다시 확보한다. 그러면 `NAVER 결과 데이터의 복제 DB`가 아니라 손품해방이 출처와 확인 이력을 가진 자체 정보가 된다.
