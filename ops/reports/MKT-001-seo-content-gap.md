- 작업 ID: `MKT-001`

- 담당 영역: 마케팅·운영
- 적용 명세: `ops/handoffs/MKT-001.md`
- 분석 범위: 현재 저장소의 공개 경로, HTML 메타데이터, `sitemap.xml`, `robots.txt`, 준비백과·가이드·커뮤니티 데이터, 관련 사업·서비스·마케팅·운영 문서
- 변경 범위: 이 보고서 신규 1개만 추가. 사이트 코드, 라우팅, 메타데이터, 콘텐츠 원본, 데이터베이스, 외부 채널은 변경하지 않음.

## 1. 작업 결과

현재 사이트에는 홈, 통합 업체 목록, 쿼리 기반 업체 상세, 비교·문의, 준비 도구, 준비백과, 커뮤니티, 업체 입점, 정책 화면이 이미 있다. 그러나 검색 유입을 받을 수 있는 구조는 `지역×행사×검색 의도`별 고유 경로가 아니라 대부분 하나의 목록 또는 상세 HTML에 쿼리 문자열을 붙이는 방식이며, 업체·글·커뮤니티 글의 개별 URL은 sitemap과 canonical 체계에 포함되지 않는다.

소스 HTML 39개와 sitemap 11개를 교차 대조했다. sitemap 항목 11개는 모두 실제 루트 HTML과 대응하지만, 모든 `<loc>`와 `robots.txt`의 Sitemap 주소가 테스트 Netlify 호스트를 사용한다. 공식 도메인은 확정되지 않았으므로 현재 호스트를 공식 주소로 판정하지 않았다.

준비백과 데이터에는 28개 글이 있고 데이터 내부의 완전 동일 제목은 없다. 다만 28개 글 모두에 정확히 같은 본문 문단 8개가 반복되며, 지역명이 들어간 제목은 0개다. 제목 기준으로 돌잔치 9개, 환갑·칠순 1개, 스몰웨딩 1개가 확인되고 백일·팔순·상견례·가족모임·기념일·생신·추모 전용 제목은 없다. 따라서 단순 글 수보다 주제 편중과 템플릿 반복을 먼저 해결해야 한다.

이 보고서의 우선순위는 검색량 추정치가 아니라 다음 근거로 정했다.

1. 확정된 전국·8개 가족행사 범위
2. 서울 돌잔치의 초기 검증 우선순위
3. 현재 경로와 내부 링크에서 확인되는 공백
4. 저장소 문서에 기록된 정적 데이터 품질 및 공개 게이트 위험
5. 공식 도메인·운영 주체·공개 기준 등 선행 결정 여부

검색량, 시장규모, 경쟁사 수, 외부 순위는 조사하거나 사실로 사용하지 않았다.

## 9. 내부 링크 구조 기준선

### 현재 사실

- 홈은 8개 행사와 17개 지역을 지원하지만 전용 landing이 아니라 `venues.html` 쿼리로 연결한다.
- `scripts/pages/home.js`가 17개 지역 링크를 동적으로 만들며 모두 같은 목록 HTML로 연결한다.
- 업체 카드는 `provider.html?id=...`, 준비백과 카드는 `article.html?slug=...`, 커뮤니티 글은 `community-post.html?id=...`로 연결한다.
- 준비백과 글은 목록 복귀, 같은 카테고리 우선 관련 글 3개, 계산기·체크리스트·업체 목록 CTA를 제공한다.
- 관련 글 선택은 먼저 같은 카테고리를 고른 뒤 배열 순서로 채우므로 지역·행사·준비 단계·의도에 따른 의미 기반 연결은 아니다.
- `guides.html`, `article-contract-questions.html`은 루트 HTML의 정적 링크 기준 인바운드가 0개다.
- `contribute.html`은 루트 정적 링크 기준 `venue.html`에서만 들어오며 `venue.html`은 301 대상이어서 실질 연결이 약하다.
- `partners.html`은 현재 주 입점 안내인 `provider-join.html`과 분리되어 있고, 일반 사용자 화면의 정적 링크 연결이 약하다.

### 개선 후보

1. `행사 허브 → 지역×행사 landing → 업체 목록/상세 → 비교·문의` 구조를 데이터가 검증된 조합부터 만든다.
2. `행사 허브 → 준비 단계·조건 가이드 → 관련 업체 필터`로 정보성 의도와 탐색 의도를 연결한다.
3. 업체 상세에서 해당 지역·행사 허브, 관련 조건 가이드, 같은 기준의 비교 목록으로 연결한다.
4. 준비백과 관련 글은 카테고리뿐 아니라 행사 유형, 준비 단계, 조건, 지역을 기준으로 선택한다.
5. 중복 정적 가이드는 대표 URL을 정한 후 합치거나 리디렉션한다. 실제 합치기·리디렉션은 별도 승인 작업이다.
6. 고아 또는 약한 경로는 유지·통합·noindex·리디렉션 중 하나를 명시적으로 결정한다.

여기서 “즉시 작성 가능”은 **외부 수치·특정 업체·미확정 가격을 쓰지 않는 내부 초안 또는 콘텐츠 브리프를 만들 수 있다**는 의미다. 실제 공개를 승인한다는 뜻은 아니다.

| 후보 | 작성 가능한 범위 | 쓰면 안 되는 내용 | 권장 처리 |
| --- | --- | --- | --- |
| 가족행사 장소 비교 전 공통 확인표 | 인원, 이동, 주차, 접근성, 식사, 취소, 외부 반입 질문 | 미확인 가격·예약 가능 여부 | 8개 행사 허브의 공통 기반 초안 |
| 환갑·칠순·팔순 장소 답사 체크리스트 | 어르신 이동, 좌석, 엘리베이터, 화장실, 주차, 식사 | 특정 업체 추천·시장 평균가 | P1 전용 글 초안 |
| 상견례 장소 계약 전 질문 | 단독룸, 좌석, 코스 시간, 소음, 주차, 예약 변경 | 인기 순위·평균 가격 | P1 전용 글 초안 |
| 스몰웨딩 확인 질문 | 보증 인원, 대관, 외부업체, 우천, 취소, 동선 | 예약·결제 기능이 이미 된다는 표현 | 기존 글 보강 초안 |
| 가족모임·기념일 장소 선택 질문 | 참석자 구성, 식사, 접근성, 공간, 시간 | 지역별 업체 수·가격 | P2 행사 허브 초안 |
| 정보 확인일·출처 읽는 법 | `basic/claimed/verified` 개념, 확인 필요 항목, 직접 재확인 | `api_collected=검증 완료` 표현 | 정책 결정 전에는 개념 초안만 작성 |

기존 `hotel-vs-restaurant-buffet`, `contract-questions`, `budget-hidden-costs`, `minimum-guarantee`, `parking-site-visit`, `vendor-verification`과 겹치는 새 글은 만들지 말고 기존 글을 우선 고유화·보강한다.

## 14. 공식 도메인 등 결정 후 가능한 작업

### `D-06` 공식 도메인·운영 주체 확정 후

1. sitemap과 robots의 테스트 호스트를 공식 도메인으로 생성한다.
2. 모든 색인 대상 유형에 절대 canonical을 적용한다.
3. 페이지별 Open Graph와 공유 이미지를 적용한다.
4. WebSite·Organization 구조화 데이터에 실제 운영 주체와 공식 URL을 사용한다.
5. 개인정보처리방침·이용약관·문의에 실제 사업자·책임자·연락 창구를 반영한다.
6. 외부 검색엔진 제출이나 Search Console 연결은 별도 외부 실행 승인 후 진행한다.

### URL·색인 정책 결정 후

1. 지역×행사 landing의 고유 경로와 기존 쿼리 URL의 canonical·리디렉션 관계를 정한다.
2. 준비백과 28개를 고유 색인 URL로 제공하고 Article·Breadcrumb 구조화 데이터를 적용한다.
3. 업체 상세 고유 URL, 폐업·병합·이전 시 리디렉션 정책을 적용한다.
4. 로그인·계정·문의·비교·등록·커뮤니티 경로별 noindex와 robots 정책을 정리한다.
5. sitemap을 정적 파일 수가 아니라 승인된 색인 대상 데이터에서 생성한다.

### 데이터·운영 결정 후

1. `D-01~D-03`과 QA-002 기준을 통과한 업체만 지역 landing과 상세 색인 후보로 사용한다.
2. `D-08`에 따라 서울 돌잔치 이후 확장 순서를 한 조합씩 정한다.
3. 추천·비교 기준, 광고 표시, 업데이트 주기를 확정한 후 비교·추천 콘텐츠를 만든다.
4. 이미지 권리와 업체별 사실 필드를 확인한 후 LocalBusiness·이미지·가격 관련 구조화 데이터를 조건부 적용한다.

## 19. 후속 작업 후보

1. **MKT-002 — 기존 준비백과 콘텐츠 품질·통합 감사**  
   28개 글의 공통 문단 제거, 주제별 고유 정보·근거·행사 차이 보강, 계약 글 대표 URL 선정안을 작성한다.
2. **PM/OPS 결정 — D-06 공식 도메인·운영 주체 확정**  
   canonical, sitemap, robots, Organization, 정책·문의 정보의 기준을 확정한다.
3. **SEO-TECH-001 — URL·색인·canonical 정보 구조 설계**  
   지역×행사, 업체, 준비백과, 커뮤니티, 로그인·전환 경로의 색인 정책과 리디렉션을 설계한다.
4. **CONTENT-001 — 서울 돌잔치 허브 콘텐츠 브리프**  
   QA-002와 공개 기준을 바탕으로 인원·지역·장소 유형·주차·단독 공간의 실제 발행 가능 범위를 정한다.
5. **CONTENT-002 — 8개 행사 콘텐츠 택소노미**  
   행사별 준비 단계, 조건, 사용자 질문, 내부 링크 필드를 정의하되 서울 돌잔치 이후 발행 순서는 D-08에 맡긴다.
6. **OPS-COPY-001 — 업체 입점·신뢰 문구 감사**  
   `provider-join.html`, `partners.html`, `about.html`, 편집 정책의 약속과 실제 기능·검수 상태를 맞춘다.
7. **SEO-DATA-001 — 검증 업체 색인 자격 정의**  
   D-01~D-03, QA-002 결과를 받아 상세·landing에 필요한 최소 필드, 신선도, 이미지 권리, 문의 가능 조건을 정한다.
8. **SEO-OPS-001 — 운영 콘텐츠 읽기 전용 인벤토리**  
   승인된 스테이징/운영 접근이 제공된 뒤 Supabase override 글·업체·배너와 소스 기준선을 대조한다.

## 21. 영향·남은 문제·병합 권장 여부
