# BIZ-002 NAVER 비의존 초기 사업·서비스 전략

- 작업 ID: `BIZ-002`
- 작성 기준일: 2026-07-22
- 결과 판정: **설계 완료 / 최종 전략·외부 실행은 APPROVAL_REQUIRED**
- 수정 범위: 이 보고서 1개만 추가
- 90일의 의미: 승인된 데이터·운영·제품 준비가 갖춰진 뒤 시작하는 **서울 돌잔치 실행 파일럿 기간**이다. 이 작업에서 실제 수집, 업체 연락, 게시, 배포, DB 변경을 실행하지 않았다.

## 1. 결론

손품해방의 전국 가족행사 범위는 유지하되, 첫 90일의 운영 역량과 검증 표본은 `서울 돌잔치`에 집중한다. 초기 상품은 업체 수를 과시하는 디렉터리가 아니라 **출처와 최근 확인일이 있는 조건을 같은 기준으로 비교하고, 문의 수신이 확인된 업체에 견적을 요청하는 정보·리드 플랫폼**이다.

공급 구조는 다음 순서다.

```text
허용된 공공데이터에서 후보 발견
  → 중복·폐업·주소 변경 점검
  → 가족행사 가능성 분류(아직 확인이 아니라 후보)
  → 업체 소유권 신청·사업자 상태 확인
  → 업체가 가격·인원·주차·시설·사진을 직접 제출
  → 관리자가 출처·권리·표준 항목 검수
  → 확인 수준과 최근 확인일을 표시해 검색·비교
  → 문의 수신 업체에만 견적 요청·응답
  → 자체 콘텐츠·자체 후기 축적
  → 품질·응답 지표를 통과한 지역×행사만 순차 확장
```

예약·결제·에스크로·정산은 이 MVP에 포함하지 않는다. `Premium 월 39,000원`과 `중개 수수료 10~15%`는 기존 미확정 제안값이며 가격·수수료·혜택을 공개하거나 확정하지 않는다.

## 2. 사실·확정·제안·승인 항목의 구분

### 2.1 확정된 방향

- 서비스 범위는 전국의 가족행사이며 서울 돌잔치는 초기 운영 우선순위다(`ADR-001`).
- 1차 전환 종점은 비교·견적 문의와 업체 응답이다. 예약·결제·에스크로·정산·결제자 후기는 후속 검토다(`ADR-011`).
- 현재 NAVER 검색 결과에 의존해 만든 공개 배열 4,960건을 검증 업체로 간주하지 않는다(`QA-009`, BIZ-002 공유 계약).
- NAVER 원문·검색 요약·파생 후기 수·추천 점수를 새 데이터 원천으로 사용하지 않는다. 같은 업체를 다시 다루려면 허용된 공공데이터, 업체 공식 채널, 업체 제출 또는 적법한 이용자 제보에서 필드를 독립적으로 확보한다.

### 2.2 현재 공식 자료에서 확인한 사실

- 지방행정인허가데이터개방시스템은 2026년 4월 16일 0시에 폐쇄됐고, 지방행정 인허가정보와 생활편의정보는 공공데이터포털에서 제공된다. 따라서 신규 설계의 기준 URL·API·이용조건은 과거 LOCALDATA가 아니라 현 공공데이터포털이어야 한다. [LOCALDATA 폐쇄 안내](https://www.localdata.go.kr/portal/end.do), [행정안전부 지방행정 인허가정보 Open API 참고자료](https://www.data.go.kr/bbs/ntc/selectNotice.do?originId=NOTICE_0000000004566)

- 공공 인허가 데이터의 업체명·주소·인허가/영업 상태는 업체 존재와 영업 상태의 후보 근거일 뿐, 돌잔치 진행 가능 여부·가격·인원·주차·시설·사진 권리·문의 수신 의사를 증명하지 않는다.

## 4. 4,960건의 재정의와 처리 원칙

### 4.1 새 사업 정의

`4,960건`은 업체 수, 입점 수, 검증 업체 수, 공개 가능 수, 전국 커버리지 수가 아니다. **출처와 이용권한을 다시 확인해야 하는 NAVER 파생 레거시 후보 묶음**이다.

- 이름: `NAVER 파생 레거시 후보` 또는 내부 상태 `legacy_source_hold`
- 공개 재고·서울 파일럿 분모·SEO 업체 수·영업 대상의 신뢰 수치에서 제외
- 원본 보존·삭제·공개 번들 격리는 QA-009/D-18/OPS-009의 별도 통제를 따르며 이 작업에서 실행하지 않음
- 같은 실체를 공공데이터에서 발견해도 기존 필드의 출처만 바꾸지 않는다. 허용된 출처에서 값을 새로 읽고 출처 ID·기준일·확인자를 붙여 독립 레코드/필드로 만든 뒤 중복 관계만 연결한다.
- 독립 재확인 전에는 가격, 전화, 이미지, 행사 가능, 후기, 평점, 문의 가능 여부를 승계하지 않는다.

### 4.2 4,960건 재분류 완료 지표

재분류 작업의 완료는 데이터 삭제나 공개를 뜻하지 않는다. 다음 집계가 서로 겹치지 않고 합계 4,960으로 재현될 때 완료다.

1. `보존·공개 중지`: 독립 출처가 아직 없는 후보
2. `독립 재발견`: 새 공공데이터/업체 제출에서 같은 실체를 확인했으나 아직 검수 전인 후보
3. `중복/폐업/이전 의심`: 사람이 판단해야 하는 후보
4. `비대상`: 가족행사 제공 가능성 분류에서 제외된 후보

`독립 재발견` 수를 검증 업체 수로 합산하지 않는다. 신뢰 단계 5 이상만 비교 가능한 공개 정보, 단계 6만 문의 가능 업체 분모로 사용한다.

## 5. 데이터 신뢰 단계와 공개 라벨

신뢰는 업체 전체에 한 번 찍는 배지가 아니라 필드마다 다른 출처·확인일을 갖는다. 업체의 사업자 상태가 확인돼도 가격·주차·사진이 자동 확인되는 것은 아니다.

| 단계 | 내부 의미 | 허용 공개 | 다음 단계 조건 |
| --- | --- | --- | --- |
| T0 `legacy_source_hold` | NAVER 파생 레거시 후보 | 공개·집계하지 않음 | 허용된 출처에서 독립 재발견 |
| T1 `public_candidate` | 공공데이터에서 이름·업종·주소·영업 상태 후보 발견 | 원칙적으로 검수 대기; 공개 시에도 `공공데이터 후보`와 기준일 | 이용허락 기록, 중복·폐업·주소 점검 |
| T2 `identity_checked` | 실체·주소·공공 영업 상태 후보가 정리됨 | 업체명·지역·업종·출처·기준일의 최소 정보 | 행사 가능성 근거 또는 업체 참여 |
| T3 `event_fit_candidate` | 분류 질문표상 가족행사 제공 가능성이 있음 | `행사 가능 확인 필요` | 업체 소유권·사업자 관계 확인 |
| T4 `owner_business_checked` | 업체 담당자 관계와 제출 사업자번호 상태를 확인 | `업체 제공 정보` 입력 주체와 확인일 | 표준 필드·사진 권리 제출 |
| T5 `field_reviewed` | 제출 필드를 관리자가 출처·권리와 함께 승인 | 승인된 필드만 `업체 제공·관리자 검수`로 공개, 결측은 명시 | 문의 수신 동의·연락 경로 E2E |
| T6 `inquiry_ready` | 문의 수신 상태와 전달·응답 경로가 확인됨 | 비교·견적 문의 대상 | 실제 문의 운영·최근성 유지 |
| T7 `experience_evidence` | 손품해방 내부 경험 후기가 검수됨 | 후기 유형·행사 시점·검수일 표시 | 예약 확인 정책은 별도 승인 |

필수 공개 라벨은 `정보 제공 주체`, `원 출처`, `데이터 기준일/최근 확인일`, `확인 필요`, `업체 제공`, `이용자 후기`다. 금지할 라벨은 근거 없는 `공식 인증`, `예약 확인`, `후기 많은`, `평점 좋은`, `추천`, `실시간 가격`이다.

### 7.5 제안 내부 통과선

아래 값은 통계적 시장 확정이나 공개 약속이 아니라, 작은 파일럿의 다음 단계 여부를 판단하기 위한 **제안값**이다. 운영 역량과 Day 30 기준선을 본 뒤 PM·사용자가 확정한다.

- 공개 품질: 공개 필드 출처·제공 주체·최근 확인일 표시 100%, 미해결 중복/폐업/사진 권리 공개 0건
- 공급 최소 표본: T6 문의 가능 업체 20곳 이상. 한 유형·한 구에 쏠리면 20곳을 넘어도 통과로 보지 않음
- 수요 최소 표본: 전달 성공한 유효 문의 30건 이상. 미달이면 전환율 결론을 내리지 않고 기간/유입 경로만 재설계
- 전달 신뢰: 문의 전달 성공률 95% 이상, 성공으로 오표시된 실패 0건
- 업체 가치: 유효 문의의 50% 이상에 업체 응답이 있고, 첫 응답시간의 중앙값을 측정 가능
- 운영 신뢰: 검수·오류·후기 큐의 90% 이상이 승인된 내부 처리기한 안에 1차 판정. 실제 기한 숫자는 인력 승인 후 결정
- 안전: 연락처 목적 외 사용, 사업자번호 공개/분석 전송, 근거 없는 예약 확인 후기, 미표시 광고 각 0건

한 개 통과선만으로 확장하지 않는다. 공개 품질·전달 신뢰·안전은 필수이며, 공급·수요 표본은 부족하면 `미검증`으로 남긴다.

## 8. 핵심 지표 체계

### 8.1 북극성 지표

**`응답을 받은 유효 견적 문의 수`**를 초기 북극성 지표로 제안한다. 단순 등록 업체 수나 문의 제출 수는 품질과 연결을 증명하지 못한다.

유효 문의의 제안 정의는 중복/테스트/스팸이 아니고, 이용자 동의·연락 가능 수단·행사 유형·희망 시점·인원·지역·필수 조건을 갖추며, 선택한 T6 업체에만 전달된 문의다. 업체가 부적합으로 판정한 사유도 보존해 정의를 조정한다.

### 8.2 퍼널과 가드레일

| 축 | 지표 |
| --- | --- |
| 데이터 공급 | 출처 등록 후보, T1→T6 전환율, 중복/폐업/이전율, 필드 완성률, 90일 이내 재확인율 |
| 업체 참여 | 연락 승인 대상→소유권 시작→승인→프로필 완성→문의 수신 전환, 승인 소요시간, 권한 회수율 |
| 고객 행동 | 검색 0건율, 목록→상세, 상세→비교, 비교→문의 시작, 시작→제출, 제출→전달 성공 |
| 문의 성과 | 유효 문의율, 열람률, 응답률, 첫 응답시간 중앙값, 거절/만료/중복/실패율 |
| 신뢰 | 출처·최근 확인일 표시율, 오류 신고율/처리시간, 오래된 공개 필드, 사진 철회, 후기 반려·신고·이의제기율 |
| 콘텐츠 | 자체 콘텐츠 공개 수보다 검색→비교 보조 클릭, 수정률, 기준일 준수, 견적 질문 활용률 |
| 안전 | 무권한 편집, 목적 외 개인정보, 미확인 예약 후기, 광고 미표시, NAVER 파생값 재승계 건수(모두 목표 0) |

문의 본문, 연락처, 사업자번호, 증빙은 분석 이벤트 속성에 넣지 않는다. 전국 숫자는 지역·행사 분류값이 존재한다는 이유만으로 준비 완료 지표로 쓰지 않는다.

## 9. 전국·행사별 확장 방식

전국은 서비스 분류와 장기 범위로 계속 노출하되, `정보가 부족한 지역/행사`를 준비 완료처럼 보이지 않게 한다. 운영 확장은 지역×행사 코호트 한 개씩 수행한다.

### 9.1 다음 코호트 점수표

| 판단축 | 확인 질문 | 통과 증거 |
| --- | --- | --- |
| 수요 | 서울 파일럿 또는 검색/제보에서 반복되는 0건·문의 수요가 있는가 | 동일 정의의 검색·문의·콘텐츠 수요 데이터 |
| 공급 | 허용된 공공데이터 후보와 참여 가능 업체가 있는가 | 이용허락 등록 원천, 예상 T3/T6 분모 |
| 정보 적합성 | 해당 행사의 비교 표준 항목을 만들 수 있는가 | 행사별 질문표와 결측 정책 |
| 운영 여력 | 서울의 검수·오류·문의·후기 큐를 훼손하지 않고 처리 가능한가 | 승인된 내부 처리기한 준수율과 담당자 용량 |
| 경제성 학습 | 문의가 업체에게 의미 있고 반복 가능성이 있는가 | 업체 응답·유효 판정·지불의향 인터뷰 |
| 안전 | 출처·권리·개인정보·광고 표시 기준을 재사용할 수 있는가 | 품질·보안 검수 PASS |

### 9.2 확장 순서 원칙

1. 서울 돌잔치에서 장소/연회·식사 중심의 비교·문의 계약을 먼저 안정화한다.
2. 같은 서울 안에서 인접 돌잔치 서비스 또는 수요가 확인된 다른 가족행사 중 하나를 선택한다.
3. 다른 지역은 동일 행사 계약을 재사용할 수 있는 코호트부터 검증한다.
4. 다음 지역·행사 우선순위는 현재 문서에서 확정하지 않는다. 검색 0건·문의·후보 품질·운영 여력 데이터를 보고 D-08 후속 결정으로 남긴다.

## 10. 업체 입점 유도 정책

### 10.1 제안 가치와 순서

1. `정확성`: 무료 소유권 확인과 기본 정보 수정, 제공 주체·갱신일 표시
2. `비교 준비`: 가격·인원·주차·시설을 같은 기준으로 입력해 반복 상담 감소
3. `적합 문의`: 지역·시점·인원·필수 조건이 있는 문의만 수신
4. `운영 학습`: 문의 열람·응답과 프로필 결측을 확인

공공데이터에 실렸다는 이유만으로 업체를 입점 업체로 부르지 않는다. 업체 페이지가 먼저 있어도 담당자 승인 전 자사 정보 입력 권한이나 문의 수신을 열지 않는다.

### 10.2 금지 원칙

- 소유권·정정권을 유료 가입과 묶지 않음
- `무료 입점 후 자동 상위 노출`, `예약 증가 보장`, `독점 문의`, 미확정 가격·혜택을 제안하지 않음
- 외부 연락 승인 전 업체 연락을 실행하지 않음
- 거절·수신거부·폐업·오승인 회수 이력을 무시하고 재접촉하지 않음

## 11. 자체 콘텐츠·후기 전략

### 11.1 자체 콘텐츠가 먼저 채울 공백

- 서울 돌잔치 준비 순서와 업체 상담 질문
- 인원·보증 인원·식대/대관/돌상 포함 범위를 읽는 법
- 주차, 유아·어르신 동선, 단독 공간, 외부업체 반입 비교법
- 견적 조건표 예시와 `실견적 아님`, 기준일, 확인 방법
- 업체 선택 전 확인할 취소·변경·우천·최소 인원 질문

콘텐츠는 특정 업체 추천을 가장한 데이터 보강 수단이 아니다. 공개 사실, 운영팀 설명, 제안, 업체 협찬을 분리한다.

### 11.2 후기의 단계

| 후기 유형 | 90일 취급 | 공개 라벨 |
| --- | --- | --- |
| 문의 경험 피드백 | 문의 당사자 여부를 내부 이벤트로 확인할 수 있을 때 검수 후 사용 | `손품해방 문의 경험` |
| 자체 이용 후기 | 로그인, 행사 시점·조건, 광고/협찬, 검수·신고·철회 절차가 있을 때 공개 | `이용자 후기` |
| 예약 확인 후기 | 현재 예약 기능이 없으므로 미제공. 오프라인 증빙 또는 업체 대조의 최소수집·보유·분쟁 정책과 제품 상태가 별도 승인된 뒤 파일럿 | 승인 전 사용 금지 |

영수증·계약서·예약자 연락처를 공개 자료로 받지 않는다. 증빙 기반 예약 확인은 법적 문서·개인정보·운영 비용을 동반하므로 수익모델이나 90일 공개 약속으로 확정하지 않는다.
