## 1. 결론

D-29의 추천 묶음은 **공식 서면 업무 채널만, 목적별 데이터 분리, 원문 최소화, HMAC 기반 중복·수신거부 대조, 지점 기준 보수적 차단, case 단위 접근, 기한 있는 보유와 fail-closed 발송**이다. 다만 이 설계는 법률 자문이나 발송 허가가 아니다.

다음 조건이 모두 충족되기 전에는 연락처 수집·저장과 발송을 각각 0으로 유지한다.

1. 사용자가 D29-1~D29-9를 항목별로 승인한다.
2. 개인정보/법률 책임자가 최종 문구와 실제 채널을 보고 정보통신망법 제50조상 광고성 정보·사전 동의·예외·표시·수신거부 요건, 개인정보 보호법상 처리 근거·고지·권리 보장, 개인사업자·담당자 연락처 취급을 확인한다.
3. D-23~D-26, 역할·문의 E2E, 운영 인력·회신·사고 대응, MKT-010 배치 승인이 모두 완료된다.
4. 후속 기술 계약과 테스트가 이 문서의 목적·필드·범위·보유·삭제를 실제로 강제한다.
5. 외부 발송을 타인에게 위탁하거나 관련 전기통신역무를 이용한다면 2026-10-01 시행 예정 제50조의3의 적용 여부와 전송자격인증·관리감독 요건을 시행 전에 다시 확인한다.

**추천 운영 대안:** 위 법률 확인에서 최초 아웃바운드가 허용된다는 명확한 결론을 얻지 못하면 D-26 대안 C인 `자발 신청만`을 택한다. 모호함을 연락 실험으로 해소하지 않는다.

필드명은 기술 스키마 확정이 아니라 후속 계약의 의미 기준이다. 실제 업체값은 이 문서에 없다.

| 금지 | open pixel, 개인 단위 열람·클릭 추적, 감정/성향/응답 가능성 점수 | 이번 목적 밖 분석이며 D-26에도 없음 |

수신함에 도착한 응답 원문은 운영 DB·티켓·채팅으로 복사하지 않는다. 승인된 업무 수신함에서 코드화 후 종결 30일 또는 수신 60일 중 빠른 날에 삭제하는 것을 추천한다. 법정 보존이나 분쟁상 필요가 확인되면 관련 최소 부분만 별도 hold로 분리한다.

### 5.3 C. 거절·수신거부 차단

| 구분 | 필드 | 이유·제약 |
| --- | --- | --- |
| 필수 | `suppression_id`, `contact_hmac`, `channel_type`, `hmac_key_version`, `normalization_version` | 같은 식별값을 원문 없이 대조. HMAC도 개인정보/가명성 있는 식별값으로 취급 |
| 필수 | `provider_id`, 정확한 `branch_id`, `scope_type`, 적용 scope ID 목록 | 식별값·지점·본사·브랜드 범위를 분리 |
| 필수 | `reason_code`, 효력 시작 시각, 원 dispatch/event ID, 접수 경로 | `do_not_contact/declined/wrong_party/disputed` 등을 구분 |
| 필수 | 요청 권한 확인 수준, scope 원문을 대체하는 제한 코드, 처리자/검수자 | 광범위 차단 요청의 권한과 뜻을 최소 형태로 재현 |
| 필수 | `review_due_at`, `delete_due_at`, `legal_hold_id` nullable, 상태 | 무기한 방치·근거 없는 보존 방지 |
| 선택 | `brand_id`, `headquarters_id`, `agency_id`와 관계 근거 ID | 명시 요청+권한 확인 때만 범위 확장 |
| 선택 | 정정 전 식별값의 별도 suppression link | 오타 정정이 과거 차단을 지우지 않게 함 |
| 금지 | 거절 메시지 원문, 사유 상세, 담당자 이름·직함·개인 연락처, 감정·민감 추론 | 차단에는 필요 없음 |
| 금지 | 공개 페이지·일반 검색·분석에서 보이는 `do_not_contact` 표시 | 비공개 운영 정보이며 업체 불이익에 쓰지 않음 |

`declined`와 `do_not_contact`는 모두 현재 파일럿의 후속 연락을 즉시 0으로 만든다. `unreachable`은 연락처 식별값만 사용 금지하며 업체 전체 거절로 추정하지 않는다. `wrong_party`는 해당 식별값을 즉시 차단하고 새 연락처 탐색을 자동 시작하지 않는다.

## 6. 원문·정규화·암호화·HMAC 선택

### 6.1 추천 구조

1. 원문: 활성 승인 배치에서 실제 발송에 필요한 업무 이메일 또는 공식 문의폼 URL만 application-level encryption으로 저장한다. 키는 데이터와 분리하고 발송 서비스만 순간 복호화한다.
2. 정규화: 발송 직전 서버 메모리에서 수행한다. 별도 평문 column·CSV·로그로 남기지 않는다.
3. HMAC: 정규화 값을 목적·환경별 비밀키로 `HMAC-SHA-256` 처리해 중복·수신거부 대조에 사용한다. HMAC key는 secret manager에 두고 DB·클라이언트·로그에 두지 않는다.
4. 일반 hash 금지: 이메일·전화번호는 후보 공간이 작아 salt 없는 SHA-256 등은 사전대입으로 되찾기 쉽다.
5. 암호화와 HMAC은 역할이 다르다. 암호화 원문은 발송 가능성, HMAC은 같은 값 대조를 위한 것이다. 어느 것도 익명화를 보장하지 않는다.

### 6.2 정규화 경계

- 이메일: 앞뒤 공백과 명백한 표기 오류만 제거하고 domain만 소문자·IDNA 표준화한다. local-part 대소문자, 점 제거, `+tag` 제거를 보편 규칙으로 적용하지 않는다.
- 문의폼: scheme/host 소문자화, 기본 port·fragment 제거까지만 한다. 지점을 구분하는 path/query를 임의 삭제하지 않는다. 세션·추적 query가 있으면 수집하지 않고 공식 고정 URL을 다시 확인한다.
- 전화·문자·메신저·SNS는 D-26 범위 밖이므로 정규화 규칙도 만들지 않는다.
- 정규화 버전이 바뀌면 기존 값과 충돌 표본을 합성 데이터로 검증하기 전 발송을 중단한다.

### 6.3 HMAC의 비익명성과 키 교체

- HMAC은 같은 입력이 같은 출력으로 연결되고, 운영자가 새 후보 입력으로 일치 여부를 시험할 수 있으므로 익명정보로 단정하지 않는다.
- 다른 목적·환경에서 같은 키를 재사용하지 않는다. 영업 suppression HMAC을 회원·문의·분석 데이터와 결합하지 않는다.
- 키 교체 뒤 원문이 이미 삭제된 suppression은 새 키로 재계산할 수 없다. 따라서 각 레코드에 key version을 두고, 후보를 모든 활성 과거 키 버전으로 메모리 계산해 대조하는 전환안을 후속 보안 검토에서 정한다.
- 키 유출 시 해당 키 버전의 HMAC은 안전한 차단값으로 신뢰하지 않는다. 발송을 중단하고 사고 범위·키 폐기·기존 suppression 복구 가능성을 개인정보/보안 책임자가 결정한다.
- 키를 영구 유지해 무기한 차단하는 것도, 키를 즉시 폐기해 재접촉을 허용하는 것도 자동 선택하지 않는다.

## 7. 업체·지점·브랜드·본사·대행사 범위

| 사건 | 기본 적용 | 확대 조건 | 자동으로 하지 않을 일 |
| --- | --- | --- | --- |
| 특정 업무 이메일의 수신거부 | 식별값 + 정확한 지점의 모든 영업 채널 | 요청자가 `우리 회사/전 지점`을 명시하고 권한·법인/브랜드 관계가 확인됨 | 다른 채널·개인 계정·본사로 우회 |
| 지점 담당자의 단순 거절 | 해당 지점, 이번 파일럿 전체 | 명시된 상위 범위가 확인된 경우 | 같은 지점을 새 주소로 재접촉, 다른 지점을 브랜드 전체 거절로 추정 |
| 본사 대표 채널의 일괄 거절 | 확인된 본사와 요청에 명시된 지점/브랜드 | 공식 조직도·계약·회신 등 최소 권한 확인 | 이름이 비슷한 독립 가맹점까지 확대 |
| 브랜드 담당자의 요청 | 명시된 브랜드와 확인된 branch 목록 | 다브랜드 권한이 별도로 확인됨 | 모회사·계열사·다른 브랜드로 자동 전파 |
| 대행사 요청 | 대행사 자신의 식별값과 명시된 대리 범위 | 고객사·지점, 기간, 연락 범위에 대한 권한 확인 | 대행사가 답했다는 이유로 고객사 전체 차단 또는 직접 연락 우회 |
| 반송/없는 주소 | 그 식별값만 `unreachable` | 없음 | 업체 거절로 표기, 문의폼·전화로 자동 전환 |
| 잘못된 담당자/퇴사자 | 해당 식별값 즉시 차단 | 새 공식 채널은 새 배치·법률·preflight 승인 | 개인에게 새 담당자 연락처 요구 |
| 인수·합병·브랜드 변경 | 기존 suppression 유지, 관계 불확실이면 발송 0 | 새 canonical 관계와 승계 범위를 사람이 확인 | suppression 삭제, 이름 변경을 신규 업체로 취급 |

범위가 서로 충돌하면 더 넓은 차단을 일단 적용하되 영구 확정하지 않고 privacy owner의 정정 case로 보낸다. 충돌이 풀릴 때까지 발송은 0이다.

## 9. 발송 직전 중복·수신거부 검사

### 9.1 fail-closed 순서

각 최초·후속 발송마다, 예약 시점과 실제 전송 직전에 모두 아래를 검사한다. 하나라도 `unknown/error/conflict/stale`이면 발송을 거부한다.

1. D-23~D-26·D-29 사용자 승인, 법률 확인, 역할·문의 E2E, 운영·사고 담당, MKT-010 batch 승인 유효성
2. 정확한 `candidate/provider_id + branch_id`, 주소, brand/headquarters/agency 관계와 출처 기준일
3. 채널이 현재 공식 서면 업무 채널인지, 개인·추정 값이 아닌지, 문의폼에 영업 금지 표시가 없는지
4. 원문을 메모리에서 승인된 버전으로 정규화하고 모든 활성 HMAC key version으로 식별값 계산
5. 식별값, 지점, 본사, 브랜드, 대행사 scope의 `do_not_contact/declined/wrong_party/disputed` 검사
6. 전 batch·전 담당자·예약 queue의 `sent/delivered/no_response_closed/unreachable`과 최초/후속 횟수·7일 간격 검사
7. 수신거부·거절·답변·분쟁·반송 뒤 예약 발송이 모두 취소됐는지 검사
8. 승인 template ID/version/digest, 발신 주체, 회신·수신거부 경로, 비용 0원, 공개 링크 안전성 검사
9. 주간·파일럿 상한, 열린 P0/P1, 적체 중단선, reviewer/approver/incident owner 가용성 검사
10. `batch + branch + channel + contact_hmac + attempt_no` idempotency key로 발송 슬롯을 원자적으로 예약한 뒤 한 번만 전송

검사 결과는 원문 없이 정책 버전·시간·allow/deny·사유 코드·내부 ID만 감사한다. 운영자가 `force send`로 우회할 수 없고, 예외가 필요하면 새 배치·사용자·법률 승인으로 돌아간다.

### 9.2 상태별 다음 행동

| 입력 상태 | 다음 행동 |
| --- | --- |
| `interested` | 자동 후속 취소; D-24 승인 비공개 경로만 안내 |
| `declined` 또는 명확한 거절 | 즉시 해당 지점 후속 0; 범위 확인 후 suppression |
| `do_not_contact` | 즉시 모든 적용 scope 예약 취소; 사유 질문 금지 |
| `wrong_party` | 식별값 차단; 다른 연락처 요구·자동 탐색 금지 |
| `unreachable` | 자동 재시도·채널 전환 금지; 다음 승인 batch에서만 정정 검토 |
| `disputed` | 대상 scope 전체 발송 중단; privacy/operations 재심 |
| `no_response_closed` | 이번 파일럿 재접촉 0; 12주 뒤 자동 해제 금지 |
| 검사 오류/timeout | `deny`; 성공으로 기록하거나 나중 전송 queue에 남기지 않음 |

## 10. 보유·재검토·삭제·정정·백업

아래 기간은 **법정 기간을 단정한 값이 아니라 좁은 파일럿용 추천값**이다. 실제 적용 전에 처리 근거, 최종 채널, 수탁·재위탁, 관계 법령상 보존·입증 필요를 개인정보/법률 책임자가 확인한다. 기간이 승인되지 않으면 수집·발송하지 않는다.

suppression의 3년은 편의상 무기한 보유하자는 뜻이 아니다. 짧게 삭제하면 재접촉 위험이 생기고 오래 보유하면 목적·필요성 문제가 생기므로, **매년 재승인 실패 시 발송부터 0으로 만들고** 개인정보/법률 책임자가 삭제와 프로그램 중단을 함께 결정한다.

### 10.2 대안 R-0: 자발 신청만

- 아웃바운드 후보 연락처와 발송 이력은 만들지 않는다.
- 자발 신청·회신 과정의 개인정보는 D-24와 별도 신청 처리방침 범위에서 처리한다.
- 수신거부 요청이 들어오면 아웃바운드 영업을 하지 않는 상태를 유지하고, 향후 새 아웃바운드 프로그램 시작 전에 기존 요청 승계 방식을 새로 승인한다.
- 법률 불확실성과 침해 면적이 가장 작지만 D-26의 아웃바운드 학습 목표는 검증하지 못한다.

### 10.3 정정·삭제 절차

1. 요청자의 권한을 위험에 비례한 최소 방법으로 확인한다. 신분증·개인 휴대전화 등 새 개인정보를 기본 요구하지 않는다.
2. 먼저 관련 예약·재시도·후속을 즉시 중지한다.
3. 정정은 기존 감사 이벤트를 덮어쓰지 않고 `superseded_by`로 새 버전을 연결한다. 오타 식별값은 재사용되지 않도록 별도 suppression을 유지한다.
4. 삭제 대상은 primary, replica, cache, temp, queue, mailbox, vendor, export, backup을 inventory로 계산한다.
5. 삭제 job은 대상별 성공/실패·재시도·검증자를 남긴다. 일부 실패를 완료로 표시하지 않는다.
6. 법적 보존은 `basis`, 승인자, 필드/case 범위, 시작일, 재검토일, 종료 조건이 모두 있을 때만 별도 격리한다. 포괄적 `분쟁 대비`는 근거가 아니다.
7. 삭제·정정 요청이 suppression 자체를 지워 재접촉을 만들 수 있으면 법률/개인정보 책임자가 권리 요구와 차단 목적을 함께 판단할 때까지 발송 0을 유지한다.

### 10.4 backup 복원

- backup은 운영 검색·발송에 직접 연결하지 않는다.
- 복원 전에 최신 deletion/suppression tombstone을 격리된 환경에서 적용한다.
- 적용 건수와 실패를 검증한 뒤에만 운영 연결을 허용한다.
- 삭제된 원문이 복원되면 재사용하지 않고 우선 재삭제한다.
- tombstone 또는 key version이 누락되면 복원 환경 전체의 영업 발송을 중단한다.

## 11. 로그·분석·개인 도구 금지

허용 로그는 `actor/service ID`, 내부 case/batch/dispatch ID, 목적, 정책 버전, allow/deny, 제한 사유 코드, 시각, 성공/실패뿐이다.

다음 값은 애플리케이션 로그, APM, 오류 추적, 분석, 세션 리플레이, data warehouse, 알림, URL/query/referrer, 파일명에 넣지 않는다.

- 원문·정규화 연락처, HMAC, 암호문, key version을 제외한 비밀 메타데이터
- 메시지 제목·본문·응답·서명·첨부·메일 헤더·문의폼 입력 내용
- 수신거부 상세 사유, 담당자 이름·직함, 개인 IP·기기값
- brand/지점의 비공개 suppression 상태와 법적 hold 내용

개인 이메일·개인 드라이브·메신저·스프레드시트·노션·로컬 파일·화면 캡처는 처리 도구가 아니다. 운영 도구가 준비되지 않았다는 이유로 임시 개인 도구를 쓰지 않고 발송 0을 유지한다.
