## 1. 결론

D-24 추천안은 **좁은 파일럿 + 원본 최소화 + 짧은 보유 + case 단위 접근**이다.

1. 업체 등록은 업체/지점명, 주소 또는 활동지역, 업종, 공식 업무 채널 같은 비민감 기본 정보부터 받는다.
2. 사업자 상태가 꼭 필요한 claim·문의 활성 단계에서만 요청자가 자발적으로 입력한 사업자등록번호를 서버 쪽 상태조회 요청 동안 일시 사용한다. 응답 또는 오류를 처리하면 번호는 즉시 폐기하고 영구 저장하지 않으며, 결과·조회 시각·source·오류만 보존한다. 재확인 때는 요청자가 번호를 다시 입력한다. 후보 업체 역조회, 대표자명·개업일자를 요구하는 진위조회, 원응답 전체 보관은 하지 않는다.
3. 담당자 관계는 공식 도메인 이메일·공개된 대표번호 회신 같은 **업무 채널 확인을 우선**한다. 관계 서류는 채널 확인이 불가능한 예외에만, 필요한 부분을 가린 사본 1개로 제한한다. 신분증은 받지 않는다.
4. `business=active`는 사업자 상태만 뜻한다. 담당자 관계, 페이지 소유권, 편집 grant, 업체 필드, 공개, 문의 활성은 각각 별도 승인한다.
5. 초기 사진은 식별 가능한 인물·미성년자가 없는 장소·시설·상품 사진만 받는 것을 추천한다. 사람 사진은 동의 문구·권한자·철회·보유 계약을 법률 검토한 뒤 별도 확대한다.
6. 초기 후기는 `self_reported`만 허용하고 증빙 업로드는 받지 않는 것을 추천한다. 증빙 확인 라벨이 꼭 필요할 때만 개인정보를 가린 최소 영수증/계약 확인본을 예외로 받는다. 현재 `reservation_confirmed`는 사용하지 않는다.
7. 사업자번호 원문은 상태조회 응답·오류 처리 직후 폐기하고 영구 저장하지 않는다. 관계 서류·사진 권리 서류·후기 증빙 등 **다른 원본**은 승인·반려·철회 후 30일, 수집일부터 최대 90일을 추천한다. 공개 사진 권리의 최소 확인 기록, grant·결정·감사 기록은 원본과 분리한다. 이 수치는 법정 보유기간을 단정한 것이 아니며 실제 적용 전에 개인정보/법률 책임자가 목적별 예외를 확인해야 한다.
8. 비공개 증빙은 공개 projection, 정적 번들, 분석 이벤트, 오류 로그, 이메일·채팅, CSV/JSON export에 절대 넣지 않는다.
9. 후속 BE-006 보안 계약과 QA-003 역할·RLS·Storage E2E가 통과하기 전 민감 입력을 켜지 않는다.

이 결정안은 개인정보 처리방침, 동의서, 사진 사용동의서 또는 법률 자문이 아니다. 사용자가 아래 항목을 승인해도 실제 수집은 4절의 수집 전 게이트를 모두 통과한 별도 구현·검수 카드에서만 시작한다.

## 4. 수집 전 게이트와 즉시 중단 조건

### 4.1 모두 충족해야 민감 입력을 켤 수 있는 게이트

- [ ] D-24 목적·항목·보유·접근·파기 선택이 사용자에게 승인됐다.
- [ ] 실제 운영 주체, 개인정보 보호책임자/담당 창구, 처리방침·동의 또는 다른 처리 근거를 개인정보/법률 책임자가 확인했다.
- [ ] 수탁자·재위탁자, 저장 위치·국외 이전 여부, 삭제·백업 계약을 확인했다.
- [ ] 사업자 상태조회는 브라우저가 아니라 승인된 서버 경로에서만 실행하고, 번호와 비밀키가 URL·로그·분석·DB·Storage·영구 queue·backup에 남지 않으며 응답·오류 뒤 즉시 폐기된다. 실패·재시도 때도 번호를 저장하지 않고 사용자가 재입력한다.
- [ ] 관계·사진·후기 등 보관이 승인된 제한 원본에는 전용 저장소, 전송구간 암호화, 저장/필드 암호화, 키 분리·회전, 짧은 signed access가 구현됐다.
- [ ] anon·회원·provider·content·operations·admin/owner의 RLS 허용·거부와 IDOR 우회가 QA-003에서 통과했다.
- [ ] 목록 마스킹, 로그·분석·오류·알림 redaction, export 거부가 검증됐다.
- [ ] 만료 계산, 철회, primary/derivative/cache/temp/backup 파기와 복구 후 재삭제가 비식별 가짜 자료로 검증됐다.
- [ ] 사고 접수·containment·평가·통지 판단 책임자와 break-glass 사후 검토가 지정됐다.
- [ ] 운영자가 업무상 꼭 필요한 case만 볼 수 있고 제출자·최종 승인자·재심자가 가능한 범위에서 분리됐다.

### 4.2 즉시 중단

다음 중 하나면 새 수집·조회·업로드와 관련 signed URL 발급을 중단한다. 기존 공개 사진은 위험 범위만 우선 숨긴다.

- 비밀키, 원문 사업자번호, 개인 연락처, 관계/후기/사진 증빙이 URL·로그·분석·공개 번들·알림에 나타남
- anon, 다른 회원, 다른 provider, content 역할이 제한 원본을 읽을 수 있음
- 삭제 요청 또는 동의 철회가 접수됐으나 대상 asset·파생본·백업 범위를 추적할 수 없음
- 보유기간·`delete_due_at`·법적 보존 근거가 없거나 만료 작업이 실패함
- 무제한/장기 signed URL, 원본 public bucket, 일괄 다운로드·export가 열려 있음
- 사업자 상태 하나로 ownership·grant·publication·inquiry가 함께 승인됨
- 운영/법률 책임자, 사고 대응 연락, 처리 근거 또는 고지 대상이 미확정임
- 테스트·지원 과정에서 실제 개인정보를 로컬 파일·스프레드시트·개인 이메일·채팅에 복사해야 함

### 5.1 자유서술과 첨부 통제

- 사업자번호·연락처·신분증을 자유서술에 적지 말라는 선행 안내와 client/server pattern 경고를 둔다.
- 경고는 저장 전 막는 보조 수단이며, 패턴이 없다고 안전하다고 판단하지 않는다.
- 파일 형식·크기·페이지 수 allowlist, 악성 파일 검사, 이미지 metadata 제거, OCR 자동 추출 기본 금지를 적용한다.
- 보완 요청은 부족한 사실과 허용 대체 수단만 알려주고 “가능한 모든 서류”를 요구하지 않는다.
- 한 파일에 여러 목적의 정보가 있으면 다시 받을 자료를 좁혀 안내하고 전체 파일을 장기 보관하지 않는다.

## 6. 공개 projection 절대 제외 목록

다음은 익명 공개 테이블/API/JSON, 정적 JS, HTML, 검색 색인, sitemap, 분석 이벤트, 클라이언트 오류, 알림 제목·본문에 넣지 않는다.

- 사업자등록번호 원문·일부·암호문·hash/token, 국세청 원응답·세부 상태 코드·과세 정보
- 요청자 user/account ID, 개인 이메일·전화, 관계 유형 상세, 재직·대행·위임 서류
- ownership/보안 위험 점수, 다른 grant 보유자, 내부 중복 점수, fraud 신호
- 사진/후기/견적 원본 증빙, private storage path, signed URL, 동의서, 인물 release
- 신고자·분쟁 당사자 정보, 관리자 메모, 자유서술, 내부 사유 상세
- 열람자 IP·기기·접속기록, 암호화 키 ID, secret, 내부 bucket·함수 경로
- 삭제·철회·법적 보존 자료와 backup 위치

공개 가능한 것은 별도로 승인된 업체 기본 assertion과 D-25가 승인한 제한적 상태 라벨·확인일뿐이다. 예를 들어 `사업자 확인`을 공개하더라도 번호, 응답, 담당자 관계, 다른 필드 검수까지 공개하거나 암시하지 않는다.

## 7. 보유·만료·철회·파기 선택안

### 7.1 추천 R-A: 짧은 파일럿

### 7.2 대안 R-B: 다른 원본의 분쟁 대응 우선

- 사업자번호: R-A와 동일하게 응답·오류 처리 직후 즉시 폐기하고 영구 저장하지 않는다. R-B로 연장할 수 없다.
- 관계 서류·사진 권리 서류·후기 증빙 등 다른 원본: case 종료 후 90일, 수집 후 최대 180일
- 파생 결정·grant·감사·접근·삭제 로그: 종료 후 3년
- 장점: 늦은 이의·사칭 재발 대조가 쉽다.
- 단점: 원본 침해 시 피해 범위, 열람·삭제 운영량, 비용과 법적 정당화 부담이 커진다.
- 판정: 초기 저용량 파일럿에는 비추천. 구체적인 분쟁 빈도와 법률 근거가 확인될 때만 선택한다.

### 7.3 수치 미승인

목적별 보유기간을 정하지 못하면 해당 원본 수집을 시작하지 않는다. 사업자번호는 기간 선택과 관계없이 즉시 폐기·영구 저장 금지가 기본이다. `운영 준비 중` 상태로 두고 비민감 등록·placeholder·`self_reported` 후기만 사용한다. 무기한 보유는 대안이 아니다.

### 7.4 만료와 파기

1. 수집 때 `retention_policy_id`, `collected_at`, `purpose_end_at`, `delete_due_at`을 결정한다.
2. 만료 전 운영 큐에는 ID·자료 종류·예정일만 보이고 원문을 목록에 노출하지 않는다.
3. 만료 시 신규 signed access와 처리 job을 막고 primary, derivative, thumbnail, temp, OCR, queue payload, export, CDN/cache를 순서대로 제거한다.
4. 삭제는 복구·재생이 어렵도록 storage 종류에 맞는 방법을 적용하고 성공·실패·재시도만 감사한다.
5. backup은 즉시 개별 삭제가 어렵다면 복원 제한과 만료 순환을 적용하고, 복원 시 삭제 tombstone을 먼저 재적용한다.
6. 삭제 실패는 완료로 표시하지 않고 Q13/보안 case로 올려 재시도와 영향 범위를 기록한다.

### 7.5 철회·삭제 요청

1. 요청자의 권한을 위험에 비례한 최소 방법으로 확인한다. 다시 신분증을 요구하지 않는다.
2. 해당 사진 공개, signed URL, 새 검수·공유·분석 이용을 우선 중단한다.
3. 철회가 business/relationship 근거를 무효화하면 관련 grant를 suspend하고 다른 업체·필드는 건드리지 않는다.
4. 원본, 파생본, cache, export, backup, 감사 기록을 구분한 범위표를 만든다.
5. 파기 가능한 항목은 승인된 일정에 따라 지우고, 유지 항목은 근거·범위·종료일을 요청자에게 설명할 수 있어야 한다.
6. 과거 승인·반려 audit를 조용히 고치거나 삭제하지 않는다. 필요한 최소 비식별 결정 기록만 남기고 원문을 분리 파기한다.

### 7.6 법적 보존 충돌

- “분쟁 가능성”만으로 포괄 보존하지 않는다. 적용 법령·명령·구체적 사건과 보존해야 할 정확한 항목을 법률 책임자가 확인해야 한다.
- hold에는 `basis`, 승인자, 대상 case/필드, 시작일, 재검토일, 종료 조건을 둔다.
- hold 자료는 일반 자료와 분리하고 공개·검수·분석·재사용을 금지하며 제한 admin/owner만 접근한다.
- 사업자번호는 기본적으로 영구 저장되지 않으므로 포괄적 legal hold 대상에 넣지 않는다. 원문 보존을 직접 요구하는 구체적 법률 근거가 확인된 예외도 별도 사용자·법률 승인 전에는 수집·보존하지 않는다.
- hold가 끝나면 원래 `delete_due_at`과 현재 목적을 다시 계산해 지체 없이 파기한다.
- 법률 근거가 미확정이면 운영자가 임의 보존하지 않고 `APPROVAL_REQUIRED`로 중단한다.

### 8.2 필수 통제

- 전송은 TLS, 제한 원본은 공개 자료와 분리한 private storage에 저장한다.
- 사업자번호는 request-scope 메모리에서만 처리하고 DB·Storage·영구 queue·로그·분석·backup에 원문·암호문·hash/token을 저장하지 않는다. 원문 보존이 불가피하다는 기술·법률 예외는 별도 사용자·법률 승인 전 금지하며, 승인되더라도 후속 계약에서 격리·암호화·키 분리·최단 보유를 새로 정한다.
- privileged 역할은 MFA와 짧은 세션, case 단위 just-in-time 권한을 사용한다. 퇴사·대행 종료·장기 미사용 신호에서 즉시 회수한다.
- signed access 기본값은 5분, 한 case·한 asset·한 사용자·읽기 전용이다. URL은 query/referrer/분석/채팅에 남기지 않고 재사용을 제한한다.
- 목록에는 사업자번호를 표시하지 않고 상태 결과·시각만 보인다. 관계 자료의 전화·이메일·파일명은 마스킹하고, 상세도 업무에 불필요한 부분은 가린 preview를 우선한다.
- 원본 다운로드, 브라우저 저장, 인쇄, clipboard, CSV/JSON export, 일괄 열람은 기본 거부한다. 예외는 목적·대상·만료·승인자와 결과 파일 파기일을 별도 감사한다.
- 다운로드 차단이나 watermark는 화면 촬영을 완전히 막지 못한다. 최소 인력, 열람 교육, 감사와 제재를 함께 적용한다.
- 로그에는 actor, case/asset의 내부 ID, 목적, allow/deny, 시각, IP/세션 위험 요약만 남기고 원문 값·파일명·본문·signed URL은 남기지 않는다.
- 악성 파일 검사와 안전한 preview 변환이 끝나기 전 운영자 브라우저에서 원본을 열지 않는다.
- break-glass는 사고 책임자만 사유와 만료를 입력해 사용하고, 사용 후 독립 검토한다. break-glass로 audit를 지울 수 없다.

## 9. 사고·오수집·분쟁 대응

### 9.1 우발적 과수집

1. 자동 공개·분석·알림·일반 큐 전달을 중단하고 해당 object 접근을 차단한다.
2. 원문을 복사하지 않고 case ID, 발견 시각, 노출 경로, 접근자만 사고 기록에 남긴다.
3. 개인정보/보안 책임자가 실제 노출·열람·전송 범위를 평가한다.
4. 목적에 불필요하면 승인된 안전 파기로 제거하고, 필요한 경우 제출자에게 더 적은 자료를 다시 요청한다.
5. 통지·신고 의무 여부와 시점은 법률 책임자가 당시 법령과 사실관계로 판단한다.

### 9.2 무권한 열람·계정 침해

1. 관련 세션, signed URL, grant, export를 최소 범위로 revoke한다.
2. 공개 사진·문의 등 계속 피해를 만드는 경로를 우선 hide/pause한다.
3. 변조 방지된 접근 로그를 보존하되 원본 증빙을 incident 메모에 복제하지 않는다.
4. 어떤 정보, 누구, 언제, 어떤 행위, 외부 유출 가능성인지를 범위화한다.
5. 비밀번호·키 회전, 계정 복구, 취약 경로 수정과 법률상 통지 판단은 승인된 사고 절차로 분리한다.

### 9.3 분쟁·이의

- 원결정자 아닌 admin이 원 case와 새 근거를 재심한다.
- 위험한 asset/grant/inquiry만 임시 중지하고 관련 없는 업체·필드는 유지한다.
- 원결정을 덮어쓰지 않고 `upheld/corrected/revoked/restored/rolled_back` 새 결정을 연결한다.
- 상대방에게 다른 사람의 연락처·서류·fraud 신호·내부 보안 통제를 공개하지 않는다.
- 분쟁이 있다는 이유만으로 모든 원본의 `delete_due_at`을 없애지 않는다. 법적 보존은 7.6의 별도 승인만 허용한다.

## 12. 법률 확인·후속 기술 계약·사용자 승인

### 12.1 법률 확인

이 보고서는 법률 문구를 확정하지 않는다. 2026-07-22 확인한 공식 기준은 목적에 필요한 최소 수집, 불필요해진 개인정보의 파기와 법령상 보존 시 분리 저장, 안전조치·접속기록, 정정·삭제·처리정지/동의 철회라는 방향을 요구한다. 실제 적용 시 시행일과 사업자 유형을 다시 확인해야 한다.

- [개인정보 보호법 제16조·제21조·제29조·제30조](https://law.go.kr/LSW/lsInfoP.do?lsiSeq=270351)
- [개인정보 보호법 제21조 조문](https://www.law.go.kr/LSW/lsLinkCommonInfo.do?ancYnChk=&chrClsCd=010202&lsJoLnkSeq=1020398651)
- [개인정보 보호법 제36조 정정·삭제 조문](https://www.law.go.kr/LSW/lsSideInfoP.do?docCls=jo&joBrNo=00&joNo=0036&lsiSeq=270351&urlMode=lsScJoRltInfoR)
- [개인정보 보호법 시행령 제30조 안전성 확보 조치](https://law.go.kr/LSW/lsLinkCommonInfo.do?chrClsCd=010202&lsJoLnkSeq=1025277165)
- [개인정보의 안전성 확보조치 기준](https://www.law.go.kr/LSW/admRulInfoP.do?admRulSeq=2100000265956&chrClsCd=010201&joNo=001200)

법률 책임자가 확인할 항목:

1. 사업자등록번호와 개인사업자 정보의 처리 근거, 고지·동의 방식, 상태조회 위탁/제공 관계
2. 관계 확인·사진 권리·후기 증빙의 항목별 필수/선택 구분과 동의 철회 효과
