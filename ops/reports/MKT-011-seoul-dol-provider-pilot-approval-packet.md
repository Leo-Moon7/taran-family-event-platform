# MKT-011 서울 돌잔치 업체 제한 파일럿 D-26 승인 패킷

- 작업 ID: `MKT-011`
- 기준일: `2026-07-22` (Asia/Seoul)
- 문서 상태: **승인안 작성 완료 / 실제 후보 추출·연락처 수집·업체 연락·외부 게시·비용 집행은 APPROVAL_REQUIRED**
- 수정 범위: 이 보고서 1개 신규 작성
- 비실행: 실제 업체·연락처 조사, 이메일·전화·문자·메신저·방문, 외부 게시·광고·비용, 제품·DB·API·라우팅·환경변수·패키지·CHG-A~C 변경
- 기간 정의: `Day 0`은 오늘이 아니라 모든 선행 게이트를 통과한 뒤 사용자가 승인한 시작일이다. 12주(84일) 제한 운영 뒤 Day 85~90은 신규 연락 없이 마감·판정만 한다.

## 2. 확정 사실·이번 제안·승인 필요 분리

### 2.1 확정 사실·방향

- 전국 가족행사 서비스 범위는 유지하고 서울 돌잔치는 첫 운영 검증 코호트다(`ADR-001`, `ADR-016`).
- NAVER 블로그·플레이스·지역검색 파생 4,960건은 `legacy_source_hold(T0)`다. 업체 수, 파일럿 분모, 후보·연락 대상, SEO 커버리지로 사용할 수 없다.
- 첫 제품 종점은 조건 비교→견적 문의→업체 응답이다. 예약·결제·에스크로·정산은 제공 범위가 아니다.
- 업체의 기본 등록·페이지 소유권 신청·기본 정보 수정은 무료이며 유료 상품 구매와 묶지 않는다.
- 등록 수락, 사업자 상태, 담당자 관계, 수정 권한, 필드 검수, 공개, 문의 활성은 각각 다른 판단이다. T5만 공개·비교 자격을 검토하고 T6만 견적 문의를 받을 수 있다.
- 초기 업체 제출은 모두 관리자 검수를 거치며, 모르는 가격·인원·주차·시설은 0·무료·불가로 채우지 않는다.

### 2.2 이 문서의 제안

- 첫 10곳, 공식 서면 업무 채널 1개, 최초 1회와 후속 1회, 주별·전체 상한
- 수신거부·중복 방지·무응답·거절·분쟁 처리
- 12주 운영과 Day 30/60/90 판단선
- 수치는 공개 약속이나 성과 보장이 아니라 D-26에서 선택할 내부 안전 상한이다.

### 2.3 `APPROVAL_REQUIRED`

- D-26의 대상·문구·채널·횟수·용량·기간·판단선과 실제 파일럿 시작
- 실제 후보 추출, 공식 연락처 수집·보관, 이메일·문의폼·전화 등 외부 연락
- D-23 원천 채택·활용신청·수집, D-24 사업자·소유권·사진/후기 증빙 개인정보 통제, D-25 공개 라벨·재확인 주기
- 공식 업무 연락처·발송 이력·수신거부 식별값의 처리 목적·최소 필드·접근 역할·보유·삭제를 정하는 별도 QA/사용자 승인. D-24의 공통 최소수집·접근·마스킹 원칙은 적용하지만 D-24가 이 외부 영업 연락 데이터 계약을 승인한 것으로 보지 않음
- 발신 주체·회신 주소·개인정보 창구·운영시간·법률 고지의 최종 확정
- 외부 게시·광고·제휴·인센티브·비용, Premium·가격·리드 과금·수수료·SLA
- 운영 DB·API·RLS·알림·배포·최종 병합

## 3. 대상 기준과 실제 후보 추출 전 조건

### 3.1 추천 1차 대상

첫 배치 10곳은 P1 장소군만 대상으로 한다.

| 우선군 | 포함 이유 | 필요한 최소 근거 |
| --- | --- | --- |
| 호텔·연회장·전문 행사장 | 인원·공간·주차·반입 조건이 장소 비교의 핵심 가설과 맞음 | 서울의 정확한 지점, 돌잔치 제공 근거, 승인 원천, 공식 업무 채널 |
| 음식점·프라이빗 다이닝 | 단독실·식사·보증 인원·주차 조건을 검증하기 좋음 | 서울의 정확한 지점, 돌잔치 제공 근거, 승인 원천, 공식 업무 채널 |

첫 10곳은 가능하면 3개 이상 자치구와 위 두 유형을 모두 포함하고, 한 자치구 또는 한 유형이 50%를 넘지 않게 한다. 이는 추천 순위·시장 대표 표본이 아니라 편중을 줄이는 내부 배치 규칙이다. 실제 분포가 불가능하면 숫자를 억지로 채우지 않고 PM에 `보류`로 반환한다.

P2인 파티룸·공간대여, 돌상·스타일링, 케이터링은 P1 첫 배치의 정보 공백과 운영 용량을 확인한 뒤 별도 배치 승인으로만 추가한다. 사진·영상·한복·헤어메이크업 등 P3는 이번 12주 범위에서 제외한다.

### 3.2 포함 조건

실제 후보는 MKT-010에서 아래 조건을 모두 확인한 경우에만 배치 명부에 넣는다.

1. D-23에서 승인된 독립 원천 또는 업체의 자발적 신청으로 발견했다.
2. 서울의 정확한 장소 지점이며, 같은 브랜드의 다른 지점과 구분된다.
3. `돌잔치 제공`은 공공 업종명이 아니라 시점이 있는 별도 근거로 확인할 대상이다. 공공 업종만 있는 경우 행사 후보(T3) 이상으로 부르지 않는다.
4. 강한 중복·주소 충돌·휴폐업 adverse signal·분쟁이 미해결 상태가 아니다.
5. 개인 연락처·추정 이메일이 아닌 업체의 공식 서면 업무 채널을 찾을 수 있다.
6. 연락 전 수신거부·기존 접촉·다른 지점/브랜드 접촉 이력을 중복 검사할 수 있다.

### 3.3 제외 조건

- NAVER 파생값만 근거인 업체와 그 값에서 얻은 전화·주소·사진·평점·후기
- 개인 휴대전화, 개인 이메일, 개인 SNS 계정, 추정한 이메일 주소
- 이용허락 미승인 원천, 권리 미확인 사진·로고·소개문을 써야만 설명할 수 있는 후보
- 동일 지점 충돌, 폐업·이전·분쟁·권리 철회가 미해결인 후보
- 이미 수신거부·거절했거나 이번 파일럿에서 무응답 종결된 업체
- 돌잔치 근거가 없는 일반 음식점·호텔·사진관, 서울 밖 장소
- 기존 무료 정정이나 공개 상태를 유료 전환과 묶어야 한다는 조건을 요구하는 후보

### 3.4 후보 추출 자체의 선행 조건

이 보고서에서는 후보나 연락처를 만들지 않았다. D-26 이후 MKT-010에서 후보를 추출하려면 먼저 다음이 필요하다.

- D-23에서 원천과 허용 필드·출처표시·중복 lineage가 승인됨
- D-24에서 승인된 공통 최소수집·case 단위 접근·마스킹·로그/분석 비노출 통제를 적용할 수 있음
- 별도 QA/사용자 승인 카드에서 **외부 영업에 쓰는** 공식 업무 연락처, 발송 이력, 수신거부 식별값 각각의 목적·최소 필드·접근 역할·보유·삭제가 승인됨. D-24의 관계 확인용 공식 업무 채널 범위를 외부 영업 연락 승인으로 확대하지 않음
- D-25에서 실제 화면에 쓰는 업체 제출 라벨과 최근성 문구가 승인됨
- 발신자 이름·운영 주체·회신/수신거부 경로·개인정보 문의 창구가 확정됨
- 법률·개인정보 책임자가 실제 문구와 채널의 발송 요건을 확인함. 이 문서는 법률 검토나 광고성 정보 전송 판단을 대신하지 않음
- 후보마다 `candidate/provider ID`, 지점, 근거와 기준일, 공식 채널, 중복·휴폐업·기존 접촉·수신거부 검사 결과를 기록할 승인된 운영 수단이 있음

| 공개 링크 안전 | 안내에 공개 페이지를 넣는 경우 NAVER 파생 격리, T5 projection, 개인정보·권리·빈 상태 회귀 PASS | 해당 링크 사용 금지. 안전한 대체 경로가 없으면 연락 금지 |
| 운영 인력 | acquisition operator, provider reviewer, 별도 approver/incident owner와 실제 운영시간 확정 | 배치 승인 금지 |
| 회신·중단 | 발신 주체, 회신함, 수신거부·거절·분쟁 담당과 당일 중단 수단 확인 | 발송 금지 |
| 배치 승인 | 배치 ID, 고유 업체 수·분포, 근거, 채널, 문구 버전, 최초/후속 예정일, 담당, 비용 0원 확인 | 배치 실행 금지 |

같은 사람이 제출·검수·승인을 모두 맡아야 하는 상황이면 자동 승인하지 않는다. 초기 10곳의 T4와 T5는 1차 검수자 외 승인권자가 100% 다시 본다.

## 5. 채널 선택과 승인용 문구

### 5.1 채널 추천

**추천안 A — 공식 서면 채널만:** 업체가 공개한 업무 이메일을 우선하고, 없으면 공식 웹사이트 문의폼을 사용한다. 같은 업체에는 둘 중 하나만 선택한다. 기록·수신거부·문구 재현이 쉽고 개인 연락 위험이 비교적 작다.

**대안 B — 자발 신청만:** 아웃바운드 연락 없이 이미 안전하게 동작하는 신청 경로로 들어온 업체만 처리한다. 연락 위험은 가장 작지만 첫 12주 안에 대상·문구 가설을 검증하기 어렵다. 외부 게시로 모집하는 것은 이 대안에 포함되지 않는다.

**보류안 C — 전화 추가:** 서면 두 번에 전화까지 더하지 않는다. 전화가 꼭 필요하면 발신번호·운영시간·통화 기록·수신거부·법률 검토·별도 배치 상한을 새 승인안으로 내야 한다. 현재 추천·승인 범위에는 전화, 문자, 메신저, SNS DM, 방문을 넣지 않는다.

### 5.5 관심 응답 뒤 운영 안내

관심 응답은 동의가 필요한 다음 단계의 시작일 뿐 T4·T5·T6 승인이 아니다.

> 관심을 알려주셔서 감사합니다. 다음 단계에서는 정확한 업체·지점을 확인하고, 승인된 범위의 담당자 관계 확인과 정보 제출 방법을 안내드립니다. 제출했다고 바로 공개되거나 견적 문의가 활성화되지는 않으며, 각 단계의 결과와 보완이 필요한 항목을 따로 알려드립니다. 민감한 자료는 일반 이메일 답장이나 자유서술로 보내지 말고 승인된 비공개 경로만 이용해 주세요.

실제 사업자번호·관계 서류·사진 권리 증빙 요청 문구와 입력 경로는 D-24 승인 및 제품 E2E를 통과한 별도 안내가 있어야 한다.

## 6. 횟수·후속 연락·수신거부·중복 방지

### 6.1 업체당 연락 규칙

```text
배치 승인
→ 공식 서면 업무 채널 1개로 최초 1회
→ 전달 성공 + 답변 없음 + 수신거부 없음일 때 7일 뒤 후속 1회
→ 후속 뒤 답변 없음: no_response_closed
```

- 업체당 영업·파일럿 안내는 전체 채널 합산 최대 2회다. 6.2의 허용된 수신거부 처리 확인은 홍보 문구 없는 운영 확인으로 별도 구분하되 주 10회 전체 발송 상한에는 포함한다. 이메일이 실패했다고 문의폼·전화·SNS로 자동 전환하지 않는다.
- 전달 실패는 `unreachable`이며 전달 성공으로 계산하지 않는다. 채널 수정이나 재연락은 다음 승인 배치에서 다시 판단한다.
- 업체가 답변하면 자동 후속을 취소한다. 관심, 거절, 수신거부, 분쟁, 잘못된 지점 응답을 각각 구분한다.
- 무응답 종결 뒤 이번 파일럿 안에서 다시 연락하지 않는다. 12주 뒤 재접촉도 자동 허용하지 않고 새 승인 대상으로 본다.
- 지점이 다르더라도 같은 본사·운영자가 일괄 수신거부를 요청하면 요청 범위의 모든 지점에 적용한다.

### 6.2 수신거부 처리

1. `수신거부`, `그만`, `연락하지 마세요`, 명확한 거절 표현을 받으면 사유를 요구하지 않는다.
2. 다음 발송 전이 아니라 **접수 즉시** 해당 업체·지점·요청 범위의 예정 연락을 취소한다.
3. 수신거부 차단에 필요한 기록은 별도 QA/사용자 승인 카드가 정한 목적·필드·접근 범위 안에서만 남긴다. 이 문서는 원문·정규화 값·hash 중 무엇을 저장할지, 어떤 필드를 필수로 할지 정하지 않는다. 메시지 본문·개인 연락처는 분석 데이터에 복사하지 않는다.
4. 확인 답장은 법률·채널 정책상 허용되고 사용자가 원할 때 `처리했습니다`라는 운영 확인 1회만 보낼 수 있다. 마케팅 문구·신청 링크를 넣지 않는다.
5. 수신거부는 무료 정정권·일반 공개 검수·신고·분쟁 처리에 불이익을 주지 않는다.
6. D-24의 공통 최소수집·접근·마스킹 통제는 그대로 적용한다. 그러나 suppression의 보유기간·식별 방식·삭제 시점은 D-24의 승인 범위가 아니며, 별도 QA/사용자 승인 카드에서 정해야 한다. 이 문서는 기간·hash·삭제 방식을 임의 확정하지 않고, 그 계약이 없으면 발송을 0으로 유지한다.

### 6.3 발송 직전 중복 검사

각 최초·후속 발송 직전에 아래 키를 함께 확인한다.

- canonical provider/candidate ID와 정확한 지점 주소
- 공식 도메인·업무 이메일·문의폼 URL의 승인된 정규화 식별값
- 같은 브랜드·본사·대행사의 일괄 수신거부 범위
- 과거 `delivered / declined / do_not_contact / no_response_closed / disputed / unreachable`
- 다른 배치·다른 담당자의 예약 발송

중복 검사 수단이 제품·DB 변경을 요구하면 이 보고서에서 만들지 않고 PM에게 `CHANGE_REQUEST`로 반환한다. 안전한 중복 검사가 불가능한 동안 접촉 상한은 0이다.

### 7.1 추천 용량안

```text
남은 승인 후보
주 5곳 상한의 잔여

```

표본이 작거나 전환율 범위가 넓으면 더 낮은 값을 사용한다. 목표 수를 맞추려고 검수 기준을 낮추거나 무응답 업체를 다시 연락하지 않는다.

### 7.2 적체 중단선

다음 중 하나면 신규 연락과 후속 연락 예약을 모두 멈춘다.

- P0 개인정보·권한·선택 외 문의·사진 권리 사고가 1건 이상 열려 있음
- P1 잘못된 ownership/grant, authoritative 폐업, 주소·지점 충돌, 문의 전달 실패 성공 오표시가 해결되지 않음
- 표준 검수 열린 패키지가 6곳을 초과함
- P2 표준 검수의 가장 오래된 case가 5영업일을 초과함
- 다음 5영업일의 관측 처리 가능 수보다 이미 들어온 case가 많음
- reviewer와 별도 approver/incident owner 중 한 역할이 비어 있음

적체가 해소돼도 자동 재개하지 않는다. 원인, 잔량, 가장 오래된 case, 다음 2주 용량을 PM이 확인한 뒤 남은 승인 배치 안에서만 재개한다.

## 9. 지표·성공·미검증·중단 기준

### 9.1 퍼널 정의

```text
eligible_candidate
→ contact_approved
→ delivered / unreachable / declined / do_not_contact / no_response_closed
→ interested
→ claim_started
→ T4 owner_business_checked
→ profile_submitted
→ T5 field_reviewed
→ inquiry_opt_in + E2E
→ T6 inquiry_ready
→ valid_inquiry_delivered
→ inquiry_responded
```

- 자발 신청과 아웃바운드를 분리 집계한다.
- 무응답·거절·수신거부·반려를 분모에서 빼지 않는다.
- 유효 문의는 테스트·스팸·중복이 아니며 사용자 동의, 행사·희망 시점·인원·지역·필수 조건이 있고 선택한 T6 업체에만 실제 전달된 건이다.
- 문의 본문·연락처·사업자번호·증빙은 분석 metadata에 넣지 않는다.

### 9.2 안전 성공 조건 — 모두 필수

| 항목 | Day 90 성공선 제안 |
| --- | --- |
| 승인 밖 연락·수신거부 재접촉·중복 발송 | 각 0건 |
| NAVER 파생값 승계·권리 불명 사진·PII/증빙 노출 | 각 0건 |
| 무권한 편집·T5 전 공개·T6 전 문의·선택 외 전달 | 각 0건 |
| 후보·연락·수신거부·검수 결정 로그 | 대상 case 100% |
| T5 공개 필드의 출처·제공 주체·기준/확인일 | 100% |
| 공개 사진 권리·scope·철회 상태 | 100% |
| unresolved P0/P1 | 0건 |

안전 조건 하나라도 어기면 T6나 문의 수가 많아도 성공이 아니다.

### 9.3 운영·학습 성공 조건 — D-26 제안값

| 시점 | 계속/성공 판단 |
| --- | --- |
| Day 30 | 첫 10곳 중 `claim_started` 1곳 이상, 승인 밖 연락·수신거부 재접촉·즉시 공개 0건 |
| Day 60 | 누적 T6 10곳 제안, 공개 출처·필드 검수·사진 권리·문의 E2E 100%, 큐 적체선 미만 |
| Day 90 | T6 20곳, 3개 이상 자치구·2개 이상 P1 유형, 한 구/유형 50% 이하 |
| Day 90 문의 | 전달 성공 유효 문의 30건 이상, 전달 성공률 95% 이상, 유효 문의 응답률 50% 이상 |
| Day 90 운영 | 내부 처리 목표 안 1차 판정 90% 이상, 열린 패키지 6곳 이하, 최고 체류 5영업일 이하 |

이는 MKT-009·BIZ-003의 승인 전 내부 제안값을 D-26에서 선택할 수 있게 정리한 것이다. 공개 SLA, 모집 약속, 성공 보장으로 쓰지 않는다.

### 9.4 `미검증`으로 끝내는 경우

- 안전 조건은 지켰지만 T6 20곳 또는 유효 문의 30건에 미달
- 3개 자치구·2개 유형 분포를 채우지 못함
- 문의 30건 미만이라 전달률·응답률 결론이 불안정함
- 제품 장애·승인 지연으로 실제 관측 기간이 줄어듦
- 자발 신청과 아웃바운드가 섞여 채널별 효과를 분리할 수 없음

미검증은 실패를 숨기는 표현이 아니라 결론을 낼 표본이 부족하다는 뜻이다. 이 경우 대량 연락, 광고, 전국 확대, 유료 상품으로 보충하지 않는다.

### 9.5 즉시 중단

- 승인받지 않은 업체·채널·문구·횟수·비용으로 연락
- 수신거부·거절 뒤 재접촉 또는 다른 채널로 우회
- NAVER 파생 데이터 재사용, 미허용 원천·개인 연락처·추정 이메일 사용
- 개인정보·사업자번호·관계/사진 증빙 노출 또는 무권한 접근
- 사진 권리 위반, 광고·제휴 미표시, 업체 동의 없는 문의 활성
- T5 전 공개, T6 전 문의, 선택하지 않은 업체 전달, 전달 실패를 성공으로 표시
- 예약·결제·매출·상위 노출·추천·최저가·미확정 가격/수수료를 약속
- 첫 10곳 모두 `claim_started=0` 또는 연속 두 배치 모두 T4/T5 진입 0
- 7.2의 적체 중단선 도달, 고위험 신고 담당 부재

### 9.6 재개

중단 원인과 영향 범위를 기록하고, 외부 노출·grant·문의·사진을 필요한 최소 범위에서 안전하게 멈춘다. 원인 제거, 영향 case 재검토, 수신거부·중복 재검사, E2E 재검증, 실제 용량 재산정 뒤 총괄 PM이 남은 배치의 재개를 판단한다. 대상·문구·채널·횟수·비용이 바뀌면 사용자 재승인이 필요하다.

## 10. 외부에 약속하지 않을 것

다음 표현은 문구·통화·온보딩·FAQ에서 0건이어야 한다.

- `즉시 등록`, `즉시 공개`, `자동 승인`, `공식 인증`, `검증 완료 업체`
- `추천 업체`, `서울 대표`, `인기`, `1위`, `후기 많은`, `평점 좋은`, `최저가`
- `무료 광고`, `상위 노출`, `독점 문의`, `고객이 기다리고 있음`, `예약·매출 보장`
- `실시간 가격`, `예약 가능`, `즉시 예약`, `예약 확정`, `결제 가능`
- Premium 가격·혜택, 구독료, 리드 과금, 광고비, 중개·예약 수수료
- 확정되지 않은 처리시간·답변시간·공개일·문의 건수 SLA
- 공공데이터 등록·사업자 상태 확인·업체 제출·관리자 필드 검수를 하나의 전체 보증으로 합친 표현

무료라고 말할 수 있는 범위는 `기본 등록·페이지 소유권 신청·기본 정보 수정 신청`뿐이다. 승인·공개·문의·성과를 뜻하지 않고, 향후 유료 상품의 가격·혜택·전환 조건을 미리 약속하지 않는다.

## 12. 주간 운영판과 기록 최소값

업체별 원기록은 승인된 최소 접근 역할만 보고, 분석판에는 사업자번호·관계 서류·개인 연락처·사진/후기 증빙·문의 본문을 넣지 않는다.

공식 업무 연락처·발송 이력·수신거부 식별값의 목적·최소 필드·접근·보유·삭제 계약은 기존 승인·백로그·작업과 중복되지 않는 **신규 작업 후보**다. 총괄 PM이 QA/개인정보 담당과 사용자 승인 범위의 새 카드를 만들고 완료하기 전 MKT-010을 시작하지 않는다. 나머지 제품·E2E·공개 링크 항목은 기존 BE-006·FE-008·QA-003/QA-012·D-22/OPS-009와 중복되므로 새 위험 ID를 만들지 않는다. 준비되지 않은 항목을 임시 스프레드시트·개인 이메일·개인 메신저로 우회하지 않는다.
