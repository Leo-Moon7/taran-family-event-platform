## 1. 총괄 결론

손품해방은 `NAVER 정보 취합형 디렉터리`가 아니라 다음 순서의 **업체 직접 참여형 비교·견적 문의 플랫폼**으로 전환한다.

```text
이용허락이 확인된 공공데이터로 업체 후보 발견·영업 상태 후보 확인
→ 중복·폐업·주소 변경·행사 가능성 분류
→ 업체 신규 등록 또는 기존 페이지 소유권 신청
→ 사업자 상태와 담당자 관계를 각각 확인
→ 업체가 가격·인원·주차·시설·사진·문의 조건 직접 제출
→ 관리자 검수와 변경 이력
→ 출처·검수·최근 확인일이 있는 정보만 비교
→ 문의 수신이 확인된 업체에만 견적 요청
→ 손품해방 자체 콘텐츠·자체 후기 축적
→ 서울 돌잔치 지표 통과 후 지역×행사 한 조합씩 전국 확장
```

예약·결제·에스크로·정산은 후속 단계다. `Premium 월 39,000원`, `수수료 10~15%`를 포함한 가격·광고·리드 과금은 미확정이며 공개 약속으로 사용하지 않는다.

공공데이터는 데이터셋마다 이용허락과 제3자 권리·출처표시가 다르다. 공공데이터포털도 제3자 권리가 포함되면 별도 이용허락을 확보해야 한다고 안내하므로 `공공데이터`라는 한 종류로 일괄 승인하지 않는다. [공공데이터포털 이용정책](https://www.data.go.kr/ugs/selectPortalPolicyView.do)

국세청 사업자등록정보 API는 사업자번호로 상태를 조회하고 대표자명·개업일자 등을 함께 입력해 진위를 확인하는 서비스다. 신규 업체 발견 수단이 아니라 **업체 제출 이후 확인 수단**으로만 설계한다. [국세청 사업자등록정보 진위확인 및 상태조회](https://www.data.go.kr/data/15081808/openapi.do)

기존 LOCALDATA 주소는 현재 종료 페이지로 전환되어 있으므로 새 설계는 개별 `data.go.kr` dataset ID를 기준으로 해야 한다. [LOCALDATA 종료 페이지](https://www.localdata.kr/portal/end.do)

2. 현재 공개 재고·서울 파일럿 분모·업체 수 홍보·SEO 숫자에서 제외한다.

6. 가격·전화·사진·행사 가능·후기·평점·문의 가능은 승계하지 않는다.
7. 재분류 합계는 `보존·공개 중지 / 독립 재발견 / 중복·폐업·이전 의심 / 비대상`으로 4,960이 재현돼야 한다.

## 5. 업체 검증 구조

신뢰는 하나의 점수가 아니라 네 축으로 관리한다.

| 축 | 내부 질문 | 사용자 표시 예 |
| --- | --- | --- |
| 출처 | 누가 어떤 근거로 값을 제공했나 | `공공데이터 등록`, `업체 직접 제출`, `이용자 제보` |
| 사업자·소유권 | 사업자 상태와 페이지 편집 관계가 확인됐나 | `사업자 확인`, `업체 권한 확인 중` |
| 필드 검수 | 관리자가 해당 값과 사진 권리를 검수했나 | `관리자 검수`, `정보 확인 중` |
| 최근성 | 언제 다시 확인했고 만료됐나 | `최근 확인 2026-07-22`, `최근 정보 확인 필요` |

사용자가 요청한 여섯 상태는 다음 의미로 제한한다.

- `공공데이터 등록`: 승인된 데이터셋의 허용 필드가 출처·기준일과 함께 등록됨. 입점·행사 가능·추천 의미 아님.
- `정보 확인 중`: 중복·주소·영업 상태·행사 가능·제출 정보 중 하나 이상 검수 중.
- `업체 직접 제출`: 승인된 업체 담당자가 제출한 값과 제출일. 손품해방의 사실 보증이 아님.
- `사업자 확인`: 제출 사업자번호의 상태/진위 입력을 확인. 담당자 권한이나 가격·사진 검증 의미 아님.
- `관리자 검수`: 표시된 필드의 출처·조건·권리를 정책에 따라 검수. `공식 인증` 표현 금지.
- `최근 정보 확인 필요`: 재확인 주기 초과·충돌·원천 변경. 가격을 숨기거나 문의를 보류할 수 있음.

기본 업체 페이지를 완성 페이지처럼 공개하려면 최소한 stable ID, 강한 중복 없음, 업체명·표준 업종·주소/활동 지역, 승인된 독립 출처, adverse 영업 신호 없음, 관리자 확인된 행사 capability, 유효한 확인일, publication 검수, 개인정보·권리 미확인 사진 없음, 화면에 표시할 출처·최근 확인일이 모두 필요하다.

직접 견적 문의는 여기에 소유권 승인, active 사업자 상태, 문의 opt-in, 업무 연락 경로, 최근 핵심 정보, RLS·생성·열람·응답·알림 E2E를 추가로 요구한다.

## 7. 업체 확보·자체 콘텐츠 전략

업체의 초기 참여 가치는 `무료 소유권 확인`, `정확한 자사 정보 표시`, `조건이 맞는 문의 수신` 세 가지다. 무료 기본 수정권을 유료 상품과 묶지 않는다.

외부 연락 전에 다음이 준비돼야 한다.

1. 승인된 서울 돌잔치 후보 분모와 연락 대상 선정 이유
2. 소유권·사업자·담당자 확인 및 개인정보·증빙 처리 정책
3. 가격·인원·주차·시설·사진 동의 표준 입력표
4. 관리자 검수 큐, 반려·보완·이의·권한 회수 절차
5. 실제 문의 전달·열람·응답·실패 E2E
6. 연락 문구, 수신거부와 재접촉 금지, 외부 연락 사용자 승인

자체 콘텐츠는 서울 돌잔치 준비 순서, 예산 구성법, 계약 질문, 인원·주차·독립 공간·식사·돌상/외부업체 비교법부터 만든다. 콘텐츠는 출처·검토일·작성/검수 상태를 가져야 한다.

후기는 단계적으로 운영한다.

- 1단계: 로그인 사용자의 자기 경험 후기, 행사 시점·이용 조건·광고 여부·관리자 검수 표시
- 2단계: 손품해방 견적 문의를 보낸 사용자의 문의 경험 피드백
- 후속: 영수증·계약서 또는 업체 대조에 필요한 개인정보·보유·이의 정책이 승인된 뒤 제한적인 이용 증빙 확인
- 보류: 실제 예약 연결이 없는 상태에서 `예약 확인 후기` 라벨 사용 금지

## 8. 데이터 자동 업데이트 구조

```text
데이터셋 권리·endpoint·schema 사전 검사
→ 멱등 수집과 허용 필드만 비공개 observation 저장
→ 정규화·개인정보 flag
→ 후보 match와 중복·주소·폐업·행사 가능 점수
→ 관리자 검수 큐
→ 승인된 field assertion
→ 공개 projection 재계산
→ 공개 API/화면
```

자동화할 수 있는 범위는 허락된 원천 수집, 중복 후보 점수, 주소·영업 상태 변경 후보, 만료 큐, 재시도·dead-letter, 오래된 가격 숨김과 adverse 신호 문의 임시 보류다.

사람이 반드시 판단할 범위는 최초 이용허락, 중복 병합·지점 분리, 폐업·이전 확정, 행사 가능, 소유권·담당자, 업체 필드·사진·후기·견적 공개, 최초 publication과 rollback이다.

제안 기본 주기는 데이터셋 명시 cadence, 사업자 상태 신규 신청/승인 직전과 문의 업체 90일, 업체명·주소·행사 조건 90일, 변동 가격 최대 90일, 핵심 정보 180일, 소유권 관계 365일 재확인이다. 실제 숫자는 BIZ-003·OPS-008과 D-25에서 확정한다.
