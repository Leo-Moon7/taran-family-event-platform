# 총괄 PM 서브에이전트 오케스트레이션

## 목적

루트 대화가 프로젝트 총괄 PM으로서 상태 확인, 작업 기획, 전문 서브에이전트 배정, 결과 회수, 독립 검수, 수정 요청, 다음 작업 생성을 반복한다. 사용자가 다른 대화에 요청문을 복사하는 방식은 예외·대체 수단으로만 유지한다.

## 전문 에이전트

| 에이전트 | 설정 | 기본 역할 |
| --- | --- | --- |
| business-product | `.codex/agents/business-product.toml` | 사업·서비스·MVP·수익·사용자 흐름 |
| frontend-design | `.codex/agents/frontend-design.toml` | UI·반응형·접근성·프런트엔드 |
| backend-data | `.codex/agents/backend-data.toml` | Supabase·API·권한·데이터 |
| marketing-operations | `.codex/agents/marketing-operations.toml` | SEO·콘텐츠·입점·문의·운영 |
| quality-security | `.codex/agents/quality-security.toml` | 빌드·테스트·회귀·보안 |
| reviewer | `.codex/agents/reviewer.toml` | 결과 독립 검수, 읽기 전용 |

프로젝트 `.codex/config.toml`은 루트 포함 최대 4개 스레드와 깊이 1을 사용한다. 따라서 전문 작업은 한 번에 최대 3개이며 전문 에이전트는 하위 에이전트를 다시 생성하지 않는다.

## 운영 사이클

### A. 현재 상태 확인

1. `AGENTS.md`, `docs/99_의사결정기록.md`, `docs/12_통합실행계획.md`를 읽는다.
2. `ops/PROJECT_BOARD.md`, `ACTIVE_WORK.md`, `BACKLOG.md`, `FILE_OWNERSHIP.md`, `DEPENDENCIES.md`, `APPROVALS.md`, `RISKS.md`, `CHANGE_REQUESTS.md`를 읽는다.
3. 최근 결과 보고서, Git 미커밋·미추적 변경, 빌드·테스트 상태를 확인한다.
4. CHG-A~C 같은 미할당 변경을 활성 작업과 분리하고 덮어쓰지 않는다.

### B. 다음 작업 선정

우선순위는 장애·빌드 실패, 개인정보·보안, 핵심 흐름, 잘못된 공개 주장, 초기 출시 필수, 데이터 품질, SEO·콘텐츠, 운영 자동화, 확장 기능 순이다.

사용자 승인 불필요, 중복 없음, 선행 완료, 명확한 완료 조건, 정확한 파일 소유권, 미할당 변경과 비충돌, 병렬 작업 간 파일·API·DB·라우팅·공통 설정 비충돌을 모두 만족하는 작업만 선택한다. 최대 3개이며 안전하지 않으면 1~2개만 선택한다.

### C. 카드와 활성 등록

- `ops/TASK_SPECS.md`에 완전한 카드를 작성한다.
- 실제 실행 직전에 `ops/ACTIVE_WORK.md`에 작업 ID, 전문 에이전트, 수정 파일, 공유 계약, 선행 작업, 상태를 등록한다.
- `ops/PROJECT_BOARD.md`를 `IN_PROGRESS`로 바꾼다.

### D. 전문 에이전트 배정

- 카드에 맞는 custom agent를 생성한다.
- 작업 ID, 목적, 근거, 허용·금지 경로, 완료 조건, 테스트, 보고 형식만 전달한다.
- 범위 밖 문제는 수정하지 말고 후보로 반환하게 한다.
- 공통 파일, API, DB, 라우팅, 환경변수, 패키지가 필요하면 작업을 멈추고 `CHANGE_REQUEST`를 반환하게 한다.
- 쓰기 작업은 서로 겹치지 않는 Worktree/브랜치를 기본으로 한다. 현재 도구가 별도 Worktree를 제공하지 않으면 읽기 전용 병렬 작업만 같은 작업 공간에서 허용한다.

### E. 결과 회수·검수

모든 결과를 기다린 뒤 루트 PM이 범위, 완료 조건, 재현 근거, 빌드·테스트, 회귀, 충돌, 통계·사실, 보안·개인정보, 문서·코드 일치를 검수한다. 중요 작업은 `reviewer`에게 독립 검수를 추가 배정한다.

판정은 `PASS`, `REVISION_REQUIRED`, `BLOCKED`, `APPROVAL_REQUIRED`, `REJECTED` 중 하나다. `PASS`만 `DONE` 처리한다.

### F. 수정 요청

`REVISION_REQUIRED`이면 같은 에이전트에 기존 카드의 미충족 사항만 후속 요청한다. 범위를 확대하지 않는다. 기본 최대 2회이며 두 번 실패하면 `BLOCKED`로 전환한다.

### G. 완료·다음 작업

`PROJECT_BOARD`, `ACTIVE_WORK`, `BACKLOG`, `DEPENDENCIES`, `RISKS`, `APPROVALS`와 결과 보고를 갱신한다. 새 문제는 중복 검사 후 후보로 등록한다. 남은 시간과 위험이 허용하면 다음 사이클을 수행한다.

## 자동 중단

사용자 정책·사업 결정, 가격·수수료, 브랜드·서비스 범위, 운영 배포·최종 병합, 외부 게시·연락·비용, 운영 DB·개인정보·법률 문서, 파괴적 변경, 주요 기능 삭제, 해결 불가 충돌, 테스트 환경 부재, 같은 작업 2회 수정 실패, 대규모 예상 밖 변경, 비밀키·운영 권한, 사용량·시간 제한이 발생하면 중단한다.

중단 보고는 완료 작업, 진행 작업, 차단 이유, 사용자 결정, 추천안, 대안과 영향, 승인 후 재개 위치를 포함한다.

## 상설 대화 방식과 비교

| 항목 | 사용자 수동 전달 | PM 서브에이전트 오케스트레이션 |
| --- | --- | --- |
| 문맥 전달 | 사용자가 복사·전달 | PM이 카드 기반 최소 문맥 전달 |
| 충돌 통제 | 대화 간 수동 확인 | PM이 ACTIVE_WORK·소유권·의존성 사전 검사 |
| 결과 회수 | 사용자가 다시 가져옴 | PM이 대기·회수·통합 |
| 수정 요청 | 새 대화에 재전달 | 같은 에이전트에 최대 2회 제한 후속 요청 |
| 감사 가능성 | 대화별로 흩어짐 | 보드·결과 보고·판정으로 일원화 |
| 대체 사용 | 서브에이전트 미지원 시 필요 | 기본 운영 방식 |

## 현재 환경의 한계와 대체

- 현재 세션은 서브에이전트 생성·메시지·대기·중단 기능을 제공한다.
- 프로젝트 custom agent TOML은 공식 지원 형식이지만 프로젝트 신뢰와 클라이언트 설정 로드가 필요하다.
- 현재 대화의 직접 생성 도구는 custom agent 이름을 명시하는 인자를 노출하지 않는다. 따라서 같은 세션의 시험은 역할 지시를 프롬프트로 전달해 오케스트레이션 자체를 검증하고, 새 세션에서 custom agent 자동 로드·선택을 추가 확인한다.
- custom agent를 선택할 수 없는 환경에서는 동일 카드와 해당 TOML의 역할 지시를 일반 서브에이전트 프롬프트에 포함한다. 수동 별도 대화는 최종 대체 수단이다.

읽기 전용 생성·회수·검수 시험은 QA-005에서 `PASS`했다. 새 루트 작업의 실제 시작 명령은 `ops/PM_START_PROMPT.md`를 사용한다.
