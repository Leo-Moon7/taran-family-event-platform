# Supabase 마이그레이션 실행 순서

새 Supabase 프로젝트에서는 저장소 루트의 `admin-schema.sql`을 먼저 한 번 실행합니다.
기존 프로젝트에서는 아래 파일을 번호 순서대로 실행합니다.

1. `001_taran_brand.sql`
   이전 브랜드 테이블과 사이트 ID를 따란 기준으로 정리합니다.
2. `002_security_hardening.sql`
   관리자 역할, 업체 소유권, 커뮤니티 검수 정책과 중복 통계 방지 규칙을 적용합니다.
3. `003_marketplace_comparison_flow.sql`
   비교함, 통합 견적 문의, 업체 등록, 체크리스트 저장과 전환 통계 구조를 추가합니다.
4. `004_provider_automation.sql`
   업체 정보 완성도, 24시간 문의 만료, 12시간 리마인드 예약, 응답률, 정보 갱신과 관리자 예외 구조를 추가합니다.

## 004 적용 후 확인할 항목

- 업체 관리 화면에서 정보 완성도와 빠진 항목이 표시되는지
- 새 견적 문의의 `expires_at`이 발송 시점부터 24시간 뒤인지
- 문의 열람 시 상태가 `viewed`로 바뀌는지
- 답변 시 업체 응답률과 평균 응답 시간이 갱신되는지
- 180일 이상 확인하지 않은 업체의 문의 수신이 중지되는지
- 관리자 운영 예외 화면에 24시간 미응답·알림 실패·정보 갱신 대상이 표시되는지

`taran_notification_jobs`는 웹 알림 예약을 안전하게 저장하는 큐입니다. 현재 파트너가 업체 관리 화면에 접속하면
도착 알림과 12시간 리마인드가 확인 처리됩니다. 이메일이나 문자 발송을 추가할 때는 브라우저에 비밀키를 넣지 말고
Supabase Edge Function 또는 별도 서버에서 이 큐를 처리해야 합니다.

운영 프론트엔드에는 `SUPABASE_ANON_KEY`만 사용합니다. `service_role` 키는 HTML, JavaScript,
`content-config.js` 등 브라우저로 전달되는 파일에 넣지 않습니다.
